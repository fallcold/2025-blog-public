([music] 大家好，下午好。嗯，我想先从一个小小的坦白开始我的分享。呃，我上线过一些我并不完全理解的代码。生成了它，测试了它，部署了它。但我说不清它到底是怎么工作的。不过话说回来，我敢打赌，你们每一个人也都做过同样的事。[applause] 所以，现在我要承认的是：我们都会发布自己并不真正理解的代码。我想带大家做一段小小的旅程，看看这件事是怎么发生的。第一，回看历史，我们会发现历史总在重演。第二，我们掉进了一个陷阱：把 easy 和 simple 混为一谈。最后，确实有解法，但前提是我们不能把思考外包出去。过去几年我在 Netflix 推动 AI 工具落地，我必须说，这种加速是真实存在的。以前要花好几天的 backlog 事项，现在几个小时就能做完；挂在待办里很多年的大型重构，也终于开始推进了。不过问题是：大型生产系统总会以意想不到的方式失败。比如最近 CloudFare 发生的事。真出问题时，你最好能理解你正在调试的代码。而现在的问题是，我们生成代码的速度和体量都太快了，我们的理解速度已经跟不上。说实话，我自己就做过：一下子生成一堆代码，看着它想，我完全不知道这玩意在干什么。但测试过了，能跑，于是我就发了。关键在于，这其实并不新鲜。每一代软件工程师最终都会撞上一堵墙：软件复杂性超过了他们可管理的能力。我们并不是第一代面对软件危机的人；我们只是第一代在这种“无限生成”规模下面对它的人。那就先退一步，看看这一切从哪开始。60 年代末、70 年代初，当时一批聪明的计算机科学家聚在一起，说：“我们正处在软件危机中。软件需求巨大，但我们跟不上，项目周期过长，进展太慢，我们做得不够好。”于是 Dystra Kano 说过一句很经典的话，他大意是：当我们只有几台弱小计算机时，编程只是个小问题；而现在有了巨大的计算机，编程也成了巨大的问题。他的意思是，硬件能力提升了一千倍，社会对软件的需求也同比例增长，于是程序员必须在“手段和目标”之间想办法，支撑更多的软件。所以这件事一直是循环出现的。70 年代我们有了 C 语言，可以写更大的系统；80 年代有了个人电脑，人人都能写软件；90 年代有了面向对象和地狱般的继承层级，感谢 Java；2000 年代有了 agile、sprint、scrum master 告诉我们该做什么，不再是瀑布；2010 年代有云、移动、devops，软件真正吞噬了世界。而今天我们有 AI：copilot、cursor、claude、codeex、gemini，随便你叫哪个。我们可以以“描述它”的速度生成代码。模式没变，但规模真的变了，现在是无限的。Fred Brooks 你们可能知道，他写过《人月神话》。1986 年他还写过一篇论文叫《No Silver Bullet》。他在文中说，不会有某个单一创新带来一个数量级的软件生产力提升。为什么？因为难点从来不是写代码的机械部分，不是语法、打字、样板代码；难点是理解真实问题并设计解决方案。没有工具能消除这个根本难度。到目前为止我们创造的每种工具和技术，都只是让机械部分更容易。核心挑战仍然一样难：到底该构建什么、应该怎么工作。如果问题不在机械层，那我们为什么一直在优化机械层？为什么有经验的工程师也会交付自己不理解的代码？我认为答案在两个经常被混用的词里：simple 和 easy。我们总把它们互换着说，但它们完全不是一回事。呃，我在 speaker dinner 上被“曝光”是个 closure 党，所以这个例子就很直观。Clojure 语言作者 Rich Hickey 在他 2011 年的演讲《Simple Made Easy》里解释过：simple 的意思是“单一折叠、单一编织、无缠绕”，每个部分只做一件事，不和其他部分纠缠；easy 的意思是“近在手边”，不费力就能拿到，copy、paste、ship。simple 讲的是结构，easy 讲的是距离。问题是，simple 不是靠许愿得到的。simple 需要思考、设计和解缠。但 easy 总能做到，你只要把东西放近一点：装个包、用 AI 生成、从 Stack Overflow 抄个方案。这是人性，我们天生会走 easy 的路。就像我说的，Stack Overflow 的答案就在那；一个“会魔法”的框架，安装即用。但 easy 不等于 simple。easy 意味着你能很快往系统里加东西；simple 意味着你能理解自己做了什么。每次我们选择 easy，都是在选择“现在更快，未来更复杂”。老实说，这种权衡以前是有效的。复杂性在代码库里积累得还不算快，我们还有时间在必要时重构、重想、重建。我觉得 AI 打破了这个平衡，因为它是终极 easy button。它把 easy 这条路变得几乎零摩擦，以至于我们连 simple 方案都不再考虑。代码瞬间出现时，谁还会认真想架构？我给你们演示一下这事怎么发生：一个简单任务，如何在我们都爱用的对话界面里，逐步演化成复杂混乱。这个例子是刻意构造的。比如我们有个 app，想加认证。我们说 add o。于是得到一个干净的 o.js 文件。迭代几次又多了个 message 文件，你觉得挺好。然后我们又说要加 OOTH，于是现在有了 OJS 和 OOTHJS。继续迭代后，session 坏了，冲突一堆。到第 20 轮时，你已经不是在“讨论”了，而是在管理一个复杂到连你自己都记不住所有约束的上下文。被放弃方案留下的 dead code；“只是让它过”修出来的测试；因为不断说“等等其实……”而拼起来的三套方案碎片。每条新指令都在覆盖原有架构模式。我们说“让 off 在这里工作”，它做到了；我们说“修这个错误”，它也做到了。坏的架构决策没有任何阻力，代码只会变形去满足你最新请求。每次交互都在选择 easy 而不是 simple，而 easy 永远意味着更多复杂性。我们其实都知道更好的做法，但当 easy 变得这么 easy 时，我们还是会走那条路。复杂性会一路复利，直到为时已晚。AI 真正把 easy 推到了逻辑极限：决定你要什么，立刻得到代码。但危险就在这里：生成代码会把你代码库中的每种模式一视同仁。agent 分析代码库时，每一行都会被当作“要保留的模式”。第 47 行那个认证检查是模式；我 2019 年可能写过的那段“行为像 GraphQL 的怪异 gRPC 代码”，也是模式。技术债不会被识别为“债”，它只会被当成更多代码。这里的根本问题是复杂性。我知道这场里我一直说这个词但没认真下定义。最好的理解方式是：它是 simple 的反面，也就是“缠绕”。当系统变复杂后，所有东西都会碰到所有东西。你改一处，就会连带影响另外十处。回到 Fred Brooks 的 no bullet 论文，他指出每个系统里有两类复杂性。第一是本质复杂性（essential complexity）：你要解决的问题本身的根本难度。用户要支付，订单要履约，这些是软件系统存在的原因。第二是偶然复杂性（accidental complexity）：一路上我们额外加进去的所有东西，补丁式 workaround、防御式代码、曾经合理的框架与抽象，都是为了“让代码能工作”而叠上去的内容。在真实代码库里，这两类复杂性无处不在，而且纠缠得很深。要把它们分开，需要上下文、历史和经验。生成式输出不会做这种区分，所以所有模式都会被持续保留。这里有个 Netflix 真实例子。我们有个抽象层，夹在大约五年前写的旧授权代码和一个新的集中式 o 系统之间。我们当时没时间重建整个应用，所以只是在中间塞了个 shim。现在有了 AI，看起来是个把代码直接重构到新系统的好机会。听上去请求很简单，对吧？实际并不。旧代码和原授权模式耦合得太紧了：权限检查织进业务逻辑、角色假设写死在数据模型里、off 调用散落在几百个文件。agent 开始重构，改了几个文件就撞到解不开的依赖，然后就失控放弃；或者更糟，它会试图保留旧系统的一些逻辑，再用新系统把它重建一遍，我觉得这也很糟。问题是它看不到“接缝”。它无法识别业务逻辑在哪里结束、off 逻辑从哪里开始。所有东西都缠在一起，导致就算信息再完整，AI 也找不到一条干净路径。当你的偶然复杂性缠到这个程度，AI 并不是最好的改进助手。我发现它只会在上面再叠更多层。我们人能分辨，至少当我们慢下来思考时能分辨。我们知道哪些模式是本质的，哪些只是几年前某个人当时的解法。我们掌握着 AI 无法自行推断的上下文，但前提是我们在开始前愿意花时间做这些区分。那到底怎么做？当你盯着一个巨大代码库时，怎么分离偶然复杂性和本质复杂性？我在 Netflix 维护的代码库大约有 100 万行 Java，其中主服务上次看大概是 500 万 tokens。没有任何我能用的 context window 装得下它。于是我最开始想：是不是可以把代码库的大块内容直接塞进 context，看看模式能不能自己浮现，看看它能不能自己搞清楚发生了什么。结果和之前授权重构一样，[clears throat] 输出又迷失在自身复杂性里。于是我被迫换一种做法：挑选要包含的内容。设计文档、架构图、关键接口等等，并花时间把组件如何交互、该遵循什么模式写成明确要求。我其实是在写规格说明。500 万 tokens 变成了 2000 字的规范。再进一步，把规范拆成精确的执行步骤。不要含糊指令，只给精准的操作序列。我发现这样生成的代码更干净、更聚焦，也更容易理解。因为我先定义清楚，再规划执行。这就是我一段时间前称作 context compression 的方法。你叫它 context engineering 或 spec-driven development 都行，名字不重要。唯一重要的是：思考和规划成为了工作的大头。下面我讲讲它在实践里怎么跑。第一阶段，research。先把信息尽量喂进去：架构图、文档、Slack 讨论串。我前面说过很多次了，本质就是把和这次改动相关的上下文尽可能带齐。然后用 agent 分析代码库，画出组件和依赖图。这不该是一次性过程。我会不断追问，比如缓存呢？失败场景怎么处理？分析错了我就纠正，缺上下文我就补。每轮迭代都在细化它的分析。这个阶段的输出是一份研究文档：现状是什么、哪些东西互相连接、你的改动会影响哪里。几小时的探索，被压缩成几分钟阅读。[snorts] 我知道今天早上 Dex 也提到了，但这里的人类检查点至关重要。这是整个流程里杠杆最高的时刻。这里抓到错误，后面就能避免灾难。接着是第二阶段。有了有效 research 之后，我们写详细的 implementation plan。要写到真实代码结构、函数签名、类型定义、数据流。你希望它详细到任何开发者都能照着做。我常把它比作“数字填色”。你应该可以把它交给最初级的工程师说：“照这个做。”如果他逐行照抄，也应该能跑。这一步里我们会做很多关键架构决策：确保复杂逻辑正确、业务要求遵循最佳实践、服务边界合理、职责分离干净，并避免不必要耦合。我们能在问题发生前看到它们，因为我们踩过坑。AI 没这个能力，它会把每个模式都当成必须满足的要求。这一步真正的魔法是评审速度。我们可以在几分钟内验证计划，并准确知道将要构建什么。要跟上代码生成速度，我们也必须以同样速度理解我们在做什么。最后是 implementation。有了清晰计划和扎实 research，这阶段本该很简单。这正是目的。AI 有清晰规格可遵循时，上下文会保持干净和聚焦。我们避免了长对话带来的复杂性螺旋。不是 50 条消息“进化”出来的代码，而是三个聚焦产物，每一步都先验证再前进。没有被放弃的方案，没有冲突模式，没有“等等其实”留下的满地 dead code。对我来说，这个方法最大的收益是：你可以把大量实现交给后台 agent，因为难的思考你已经提前做完了。它可以直接开工实现，你去做别的，回来评审即可。而且评审很快，因为你只是在核对是否符合计划，而不是猜它又发明了什么。关键点是：我们不是让 AI 替我们思考。我们是让它加速机械环节，同时保持我们对系统的理解能力。research 更快，planning 更完整，implementation 更干净。但思考、综合和判断仍然在我们手里。还记得前面那个 AI 搞不定的授权重构吗？现在我们其实已经在做了，也开始有不错进展。关键不是我们找到了更好的 prompt。我们发现自己甚至不能直接跳到 research、planning、implementation 这些阶段。我们必须先手工做这次改动。没有 AI，只是读代码、理解依赖、改动并观察哪里会坏。那次手工迁移说实话很痛苦，但非常关键。它揭示了所有隐藏约束、必须保持的 invariant，以及 off 变化时哪些服务会炸。这些都不是再多代码分析就能自动暴露出来的。然后我们把那次手工迁移的 pull request 喂回 research 流程，作为后续研究的 seed。AI 这才看到了“干净迁移”是什么样。问题是每个实体又略有不同，所以我们还得持续盘问它：这个怎么办？有些是加密的，有些不是。我们每次都得通过多轮迭代补充额外上下文。到那之后，且仅在那之后，我们才能生成一个“可能”一次成功的计划。这里的关键词就是“可能”。我们仍在验证、仍在调整、仍在发现边界情况。三阶段方法不是魔法。它之所以有效，是因为我们先手工做过一次迁移。我们必须先赚到那份理解，才能把它编码进流程里。我依然认为不存在 silver bullet。不是更好的 prompt，不是更好的模型，甚至也不只是更好的 spec；真正的核心还是：你得足够深入地理解自己的系统，才能安全地改它。那为什么还要走这整套流程？为什么不直接和 AI 一直迭代到能跑为止？模型最终不都会强到自动搞定吗？对我来说，“能跑”不够。通过测试的代码，和能在生产里活下来的代码，不是一回事。今天能工作的系统，和未来别人还能改得动的系统，也不是一回事。真正的问题是认知差距。AI 几秒就能生成几千行代码，而理解它可能要你几个小时，复杂点可能几天，缠得更深也许永远理解不了。还有一点我觉得现在没多少人在谈：每次我们为了追上生成速度而跳过思考，我们不仅是在增加自己不理解的代码，我们还在失去识别问题的能力。那个会提醒你“这正在变复杂”的直觉，在你不理解自己系统时会逐渐萎缩。[snorts] 模式识别来自经验。我能看出危险架构，是因为我真的凌晨 3 点爬起来给它擦过屁股。我坚持更简单方案，是因为我维护过别人留下的反面方案。AI 只会生成你要求的东西，它不会编码你从失败中学到的教训。三阶段方法在弥合这个鸿沟。它把理解压缩成可评审的工件，让我们以生成的速度去审阅它。没有它，我们只是在以快于理解速度的方式累积复杂性。AI 改变了我们写代码的一切。但老实说，我不认为它改变了软件为何会失败这件事。每一代都经历自己的软件危机。Dystra 那一代通过建立软件工程这门学科应对了危机。现在我们这一代面对的是无限代码生成。我不认为答案是再来一个工具或方法论。答案是记住我们一直都知道的事：软件是人的事业。最难的部分从来不是把代码敲出来，而是先知道该敲什么。真正能长期胜出的开发者，不是生成代码最多的人，而是那些真正理解自己在构建什么的人，是那些还能看见接缝、还能识别“自己正在解决错误问题”的人。那仍然是我们。也只会是我们。最后我想留一个问题。我觉得问题不是“我们会不会用 AI”，这已经是既定事实，船已经开走了。对我来说真正的问题是：当 AI 在写我们大部分代码时，我们是否仍然理解我们自己的系统。谢谢大家。[applause] [music] [music] >> [music])
