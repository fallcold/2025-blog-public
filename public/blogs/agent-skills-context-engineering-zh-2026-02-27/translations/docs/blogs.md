TL;DR

智能体要完成任务需要上下文。上下文工程是在智能体轨迹的每一步，把恰到好处的信息放入上下文窗口的艺术与科学。在这篇文章中，我们通过回顾多种流行智能体和论文，拆解了上下文工程的一些常见策略：写入、选择、压缩与隔离。然后我们会解释 LangGraph 如何被设计为支持这些策略！

另外，也可以在这里观看我们关于上下文工程的视频。

上下文工程的一般类别

Context Engineering

正如 Andrej Karpathy 所说，LLM 像一种新型操作系统。LLM 像 CPU，而它的上下文窗口像 RAM，充当模型的工作内存。和 RAM 一样，LLM 的上下文窗口容量有限，无法容纳所有上下文来源。也正如操作系统会筛选哪些内容进入 CPU 的 RAM，我们也可以把“上下文工程”看作发挥类似作用。Karpathy 对此总结得很好：

[上下文工程是]“……在下一步中，用恰到好处的信息填充上下文窗口的一门精细艺术与科学。”

LLM 应用中常见的上下文类型

构建 LLM 应用时，我们需要管理哪些上下文类型？上下文工程可以作为一个总框架，适用于几种不同的上下文类型：

Instructions – 提示词、记忆、少样本示例、工具描述等

Knowledge – 事实、记忆等

Tools – 来自工具调用的反馈

面向智能体的上下文工程

今年，随着 LLM 在推理和工具调用方面不断提升，人们对智能体的兴趣大幅增长。智能体会交错执行 LLM 调用与工具调用，而且常用于长时任务。智能体交错执行 LLM 调用与工具调用，并利用工具反馈来决定下一步。

智能体交错执行 LLM 调用与工具调用，并利用工具反馈来决定下一步

然而，长时任务和不断累积的工具调用反馈意味着智能体往往会消耗大量 token。这会带来许多问题：可能超过上下文窗口大小、成本/时延暴涨，或导致智能体性能下降。Drew Breunig 很好地总结了更长上下文会导致性能问题的几种具体方式，包括：

上下文污染（Context Poisoning）：幻觉内容进入上下文

上下文分散（Context Distraction）：上下文压过了训练信号

上下文混淆（Context Confusion）：多余上下文影响响应

上下文冲突（Context Clash）：上下文的不同部分相互矛盾

工具调用产生的上下文会在多个智能体轮次中不断累积

基于这一点，Cognition 强调了上下文工程的重要性：

“Context engineering”……本质上是构建 AI 智能体工程师的头号工作。

Anthropic 也讲得很明确：

智能体常常会进行跨越数百轮的对话，因此需要精细的上下文管理策略。

那么，如今大家是如何应对这一挑战的？我们把智能体上下文工程的常见策略分为四类：写入、选择、压缩与隔离，并结合一些流行智能体产品与论文给出示例。然后我们会解释 LangGraph 如何支持这些策略！

上下文工程的一般类别

写入上下文（Write Context）

写入上下文，指把信息保存在上下文窗口之外，以帮助智能体执行任务。

暂存笔记（Scratchpads）

人类解决任务时，会记笔记，并记住信息以便未来相关任务使用。智能体也在获得这些能力！通过“scratchpad”记笔记，是在智能体执行任务时持久化信息的一种方法。核心思想是把信息保存在上下文窗口之外，让智能体仍可访问。Anthropic 的多智能体研究者给出了一个清晰示例：

LeadResearcher 会先思考方法并把计划保存到 Memory 中以持久化上下文，因为如果上下文窗口超过 200,000 tokens 就会被截断，而保留计划很重要。

Scratchpad 可以有几种实现方式。它可以是一个简单写文件的工具调用；也可以是会话期间持久存在的运行时状态对象中的一个字段。无论哪种方式，scratchpad 都能让智能体保存有用信息，帮助其完成任务。

记忆（Memories）

Scratchpad 有助于智能体在单次会话（或线程）内完成任务，但有时智能体也需要跨多次会话记住事情！Reflexion 提出了在每轮智能体执行后进行反思并复用这些自生成记忆的思路。Generative Agents 则会周期性地从过去的智能体反馈集合中合成记忆。

LLM 可用于更新或创建记忆

这些概念已经进入了 ChatGPT、Cursor 和 Windsurf 等流行产品，它们都具备机制，能够基于用户与智能体交互自动生成可跨会话持久化的长期记忆。

选择上下文（Select Context）

选择上下文，指把信息拉入上下文窗口，以帮助智能体执行任务。

Scratchpad

从 scratchpad 选择上下文的机制取决于 scratchpad 的实现方式。如果它是工具，智能体可以直接通过工具调用读取；如果它是智能体运行时状态的一部分，开发者可以在每一步决定向智能体暴露哪些状态内容。这为后续轮次向 LLM 暴露 scratchpad 上下文提供了细粒度控制。

Memories

如果智能体有保存记忆的能力，它们也需要选择与当前任务相关记忆的能力。这在多方面都很有用。智能体可能会选择少样本示例（情景记忆）作为期望行为示例，选择指令（程序性记忆）来引导行为，或选择事实（语义记忆）作为任务相关上下文。

确保能够选到相关记忆是一项挑战。一些流行智能体只使用固定且较小的一组文件，并始终把它们拉入上下文。例如，很多代码智能体使用特定文件来保存指令（“procedural”记忆），有时也保存示例（“episodic”记忆）。Claude Code 使用 CLAUDE.md。Cursor 和 Windsurf 使用 rules 文件。

但是，如果智能体存储的是更大规模的事实和/或关系集合（例如语义记忆），选择就更难。ChatGPT 就是一个典型例子：它会在大量用户特定记忆中进行存储与选择。

嵌入和/或知识图谱常用于记忆索引以辅助选择。尽管如此，记忆选择仍然困难。在 AIEngineer World’s Fair 上，Simon Willison 分享了一个选择失误案例：ChatGPT 从记忆中取出他的位置信息，并意外注入到他请求生成的图片里。这类意外或不想要的记忆检索，会让一些用户觉得上下文窗口“已不再属于自己”！

Tools

智能体会使用工具，但如果给得太多就可能过载。这往往是因为工具描述重叠，导致模型混淆该选哪个工具。一种方法是对工具描述应用 RAG（retrieval augmented generation），只检索与任务最相关的工具。近期一些论文表明，这能把工具选择准确率提高 3 倍。

Knowledge

RAG 是一个很丰富的话题，也可能成为上下文工程的核心挑战。代码智能体是 RAG 在大规模生产中的最佳示例之一。Windsurf 的 Varun 很好地概括了其中一些挑战：

代码索引 ≠ 上下文检索……[我们在做索引与嵌入检索……[使用] AST 解析代码并沿语义边界切块……随着代码库规模增大，嵌入检索作为启发式手段会变得不可靠……我们必须依赖 grep/文件搜索、基于知识图谱的检索，以及……一个按相关性对[上下文]重排的步骤。

压缩上下文（Compressing Context）

压缩上下文是指仅保留执行任务所需的 token。

上下文摘要（Context Summarization）

智能体交互可能跨越数百轮，并伴随高 token 消耗的工具调用。摘要是应对这些挑战的常见方式之一。如果你用过 Claude Code，你一定见过这个机制。Claude Code 在上下文窗口使用率超过 95% 后会执行“auto-compact”，并总结用户与智能体交互的完整轨迹。这类沿智能体轨迹进行的压缩可以使用多种策略，例如递归摘要或分层摘要。

可应用摘要的一些位置

在智能体设计的特定节点加入摘要也很有用。例如，可用于对某些工具调用（如高 token 消耗的搜索工具）进行后处理。再比如，Cognition 提到可以在智能体与智能体的边界处做摘要，以减少知识交接时的 token 消耗。如果需要保留具体事件或决策，摘要会变得更具挑战。Cognition 为此使用了微调模型，这也说明这一步可能需要投入大量工作。

上下文裁剪（Context Trimming）

摘要通常是用 LLM 提炼最相关的上下文片段，而裁剪则常常是过滤，或者像 Drew Breunig 所说，对上下文进行“prune（修枝）”。这可以采用硬编码启发式方法，例如从消息列表中删除较旧消息。Drew 还提到了 Provence，一个用于问答任务的训练型上下文修剪器。

隔离上下文（Isolating Context）

隔离上下文是指将上下文拆分开来，以帮助智能体执行任务。

多智能体（Multi-agent）

最流行的上下文隔离方式之一，是把上下文拆分到多个子智能体中。OpenAI Swarm 库的一个动机就是关注点分离：由一个智能体团队处理特定子任务。每个智能体都有特定工具集、指令以及自己的上下文窗口。

将上下文拆分到多个智能体

Anthropic 的多智能体研究者为此提供了有力依据：拥有隔离上下文的多个智能体优于单智能体，主要因为每个子智能体的上下文窗口都可分配给更窄的子任务。正如博客所说：

[Subagents operate] in parallel with their own context windows, exploring different aspects of the question simultaneously.

当然，多智能体也有挑战，包括 token 消耗（例如 Anthropic 报告可达聊天模式的 15×）、需要精细的提示词工程来规划子智能体工作，以及子智能体之间的协调。

通过环境进行上下文隔离（Context Isolation with Environments）

HuggingFace 的 deep researcher 展示了另一个有趣的上下文隔离案例。多数智能体使用工具调用 API，返回 JSON 对象（工具参数），可传给工具（例如搜索 API）以获取工具反馈（例如搜索结果）。HuggingFace 使用的是 CodeAgent，它会输出包含所需工具调用的代码，然后在沙箱中运行。工具调用中被选中的上下文（如返回值）再传回给 LLM。

沙箱可以把上下文与 LLM 隔离。

这样就能在环境中把上下文与 LLM 隔离开。Hugging Face 指出，这尤其适合隔离高 token 对象：

[Code Agents allow for] a better handling of state … Need to store this image / audio / other for later use? No problem, just assign it as a variable in your state and you [use it later].

状态（State）

值得特别指出的是，智能体的运行时状态对象本身也是很好的上下文隔离手段。这与沙箱化可实现相同目标。状态对象可设计为带 schema 的结构，其中包含可写入上下文的字段。schema 中某个字段（如 messages）可在每轮暴露给 LLM，而其他字段中的信息则可被隔离起来，以便更有选择地使用。

借助 LangSmith / LangGraph 的上下文工程

那么，如何应用这些思路？开始前有两个基础点很关键。第一，确保你有办法查看数据并跟踪智能体的 token 使用情况。这有助于判断应把上下文工程投入在哪些位置。LangSmith 非常适合智能体追踪/可观测性，并提供了很好的实现方式。第二，确保你有简便的方法测试上下文工程是在损害还是提升智能体性能。LangSmith 支持智能体评估，可用于测试任何上下文工程措施的效果。

写入上下文

LangGraph 设计时同时支持线程级（短期）和长期记忆。短期记忆使用 checkpointing，在智能体所有步骤之间持久化状态。这作为“scratchpad”非常实用，让你可以把信息写入状态，并在轨迹任一步读取。

LangGraph 的长期记忆让你可以在智能体的多个会话间持久化上下文。它很灵活，既可以保存少量文件（如用户画像或规则），也可以保存更大的记忆集合。此外，LangMem 提供了一组有用抽象，帮助进行 LangGraph 的记忆管理。

选择上下文

在 LangGraph 智能体的每个节点（步骤）中，你都可以获取状态。这让你能够细粒度控制每个步骤向 LLM 提供哪些上下文。

另外，LangGraph 的长期记忆在每个节点内都可访问，并支持多种检索（如取文件，以及在记忆集合上进行基于嵌入的检索）。若想了解长期记忆概览，可看我们的 Deeplearning.ai 课程。若想从具体智能体切入记忆实践，可看我们的 Ambient Agents 课程。它展示了如何在一个可长期运行、能管理邮箱并从你反馈中学习的智能体中使用 LangGraph 记忆。

具备用户反馈与长期记忆的邮件智能体

在工具选择方面，LangGraph Bigtool 库是对工具描述执行语义搜索的好方法。这有助于在大型工具集合中为任务选择最相关工具。最后，我们也提供了多个教程和视频，展示如何在 LangGraph 中使用不同类型的 RAG。

压缩上下文

因为 LangGraph 是低层编排框架，你会将智能体组织为一组节点，定义每个节点中的逻辑，并定义在节点间传递的状态对象。这种控制力提供了多种上下文压缩方式。

一种常见方法是把消息列表作为智能体状态，并借助一些内置工具定期做摘要或裁剪。不过你也可以通过多种方式加入逻辑，对工具调用或智能体工作阶段做后处理。你可以在特定位置加入摘要节点，也可以在工具调用节点里加入摘要逻辑，从而压缩特定工具调用的输出。

隔离上下文

LangGraph 围绕状态对象设计，允许你定义状态 schema，并在每个智能体步骤访问状态。例如，你可把工具调用产生的上下文存入状态中的特定字段，在真正需要之前将其与 LLM 隔离。除状态外，LangGraph 还支持使用沙箱进行上下文隔离。可查看这个仓库中的示例：一个使用 E2B 沙箱执行工具调用的 LangGraph 智能体。也可看这个视频：使用 Pyodide 做沙箱化并持久化状态。LangGraph 还对多智能体架构提供了很多支持，如 supervisor 和 swarm 库。你可通过这些视频了解更多 LangGraph 多智能体实践。

结论

上下文工程正逐渐成为智能体构建者应当掌握的一门技艺。这里我们覆盖了当下许多流行智能体中的几种常见模式：

写入上下文 - 将其保存在上下文窗口之外，帮助智能体执行任务。

选择上下文 - 将其拉入上下文窗口，帮助智能体执行任务。

压缩上下文 - 仅保留执行任务所需的 token。

隔离上下文 - 将上下文拆分开来，帮助智能体执行任务。

LangGraph 让这些策略都很容易实现，而 LangSmith 提供了便捷方式来测试智能体并跟踪上下文使用。二者结合（LangGraph 与 LangGraph）形成了一个正向反馈闭环：识别最值得应用上下文工程的机会、实施、测试、再重复。

Manus 中的上下文工程

2025 年 10 月 15 日

Lance Martin

为什么需要上下文工程

本周早些时候，我与 Manus 联合创始人兼 CSO Yichao “Peak” Ji 做了一场线上研讨会。你可以在这里看视频、这里看我的幻灯片、这里看 Peak 的幻灯片。下面是我的笔记。

Anthropic 将智能体定义为这样一种系统：LLM 主导自身流程与工具使用，并保持对任务完成方式的控制。简而言之，就是 LLM 在循环中调用工具。

Manus 是最受欢迎的通用消费级智能体之一。一个典型 Manus 任务会使用 50 次工具调用。没有上下文工程时，这些工具调用结果会在 LLM 上下文窗口中不断累积。随着上下文窗口被填满，很多人观察到 LLM 性能会下降。

例如，Chroma 有一项关于 context rot 的优秀研究，Anthropic 也解释了不断增长的上下文如何耗尽 LLM 的注意力预算。因此，在构建智能体时，谨慎管理进入 LLM 上下文窗口的内容非常重要。Karpathy 对此说得很清楚：

上下文工程是在（智能体轨迹中的）下一步里，用恰到好处的信息填充上下文窗口的一门精细艺术与科学

上下文工程方法

每个 Manus 会话都使用一个专用的云端虚拟机，为智能体提供一台带文件系统、可导航工具，以及能在该沙箱环境中执行命令（例如内置工具和标准 shell 命令）的虚拟计算机。

在这个沙箱里，Manus 使用三种主要的上下文工程策略，与 Anthropic 在这里覆盖的思路一致，也与我在许多项目中看到的实践一致：

减少上下文

卸载上下文

隔离上下文

上下文缩减

Manus 中的工具调用有“full”和“compact”两种表示。full 版本包含工具调用的原始内容（如完整搜索结果），存储在沙箱中（如文件系统）。compact 版本则保存指向 full 结果的引用（如文件路径）。

Manus 会对较旧（“stale”）的工具结果做压缩。这本质上是用 compact 版本替换 full 结果。这样智能体在需要时仍可取回完整结果，同时通过移除智能体已用于决策的“陈旧”结果来节省 token。

更新的工具结果会保持 full 形态，以引导智能体的下一次决策。这似乎是一种普遍有用的上下文缩减策略，而且我注意到它与 Anthropic 的 context editing 功能很相似：

Context editing automatically clears stale tool calls and results from within the context window when approaching token limits. As your agent executes tasks and accumulates tool results, context editing removes stale content while preserving the conversation flow, effectively extending how long agents can run without manual intervention.

当压缩收益开始递减时（见下图），Manus 会对轨迹做摘要。摘要基于完整工具结果生成，且 Manus 用 schema 定义摘要字段。这会为任意智能体轨迹生成一致的摘要对象。

上下文隔离

Manus 对多智能体采取务实做法，避免拟人化分工。人类会按角色（设计师、工程师、项目经理）组织，是因为认知限制；而 LLM 未必有同样约束。

基于这一点，Manus 中子智能体的首要目标是隔离上下文。例如，当有任务要做时，Manus 会把该任务分配给拥有独立上下文窗口的子智能体。

Manus 的多智能体由一个规划器分配任务、一个知识管理器审查对话并决定哪些内容应保存到文件系统，以及一个执行子智能体执行规划器分配的任务。

Manus 最初使用 todo.md 进行任务规划，但发现大约三分之一动作都花在更新 todo 列表上，浪费了宝贵 token。随后他们改为专门的规划器智能体，由其调用执行子智能体完成任务。

在最近一档播客中，Anthropic 的多智能体研究员 Erik Schluntz 提到他们也采用类似设计：由规划器分配任务，并把 function calling 作为启动子智能体的通信协议。Erik 以及 Walden Yan（Cognition）都提出了一个核心挑战：规划器与子智能体之间的上下文共享。

Manus 通过两种方式处理这个问题。对于简单任务（例如离散任务，规划器只需要子智能体输出），规划器会直接创建指令，并通过函数调用传给子智能体。这类似 Claude Code 的 task 工具。

对于更复杂任务（例如子智能体需要写入规划器也会使用的文件），规划器会把自己的完整上下文共享给子智能体。子智能体仍拥有自己的动作空间（工具）和指令，但会收到规划器可见的全部上下文。

在两种情况下，规划器都会定义子智能体输出 schema。子智能体有 submit results 工具来填充该 schema 后再返回给规划器；Manus 还使用受约束解码，确保输出符合定义 schema。

上下文卸载

工具定义

我们常希望智能体能执行广泛动作。我们当然可以把大量工具绑定给 LLM，并提供详细使用说明。但工具描述会消耗宝贵 token，而大量（且常常重叠或含糊）的工具也会让模型混淆。

我观察到一个趋势：智能体会使用一组小而通用的工具，让智能体获得对计算机的访问。例如，仅靠 Bash 工具和少量文件系统访问工具，智能体就能执行非常多样的动作！

Manus 把这理解为分层动作空间：函数/工具调用层 + 虚拟计算机沙箱层。Peak 提到 Manus 使用一组小规模（< 20）的原子函数，包括 Bash 工具、文件系统管理工具和代码执行工具。

Manus 不会把函数调用层越堆越大，而是把大多数动作卸载到沙箱层。Manus 能通过 Bash 工具在沙箱中直接执行许多实用工具；MCP 工具也通过 CLI 暴露，智能体同样可以用 Bash 工具执行。

Claude 的 skills 功能采用了类似思路：skills 存在文件系统中，而不是作为绑定工具；Claude 只需少量简单函数调用（Bash、文件系统）就能逐步发现并使用它们。

渐进式披露（progressive disclosure）是让 Agent Skills 既灵活又可扩展的核心设计原则。它像一本组织良好的手册：先目录、再具体章节、最后详细附录。skills 让 Claude 只在需要时加载信息……拥有文件系统与代码执行工具的智能体，在处理具体任务时不必把整份 skill 全部读入上下文窗口。

工具结果

由于 Manus 能访问文件系统，它也能卸载上下文（例如工具结果）。如上所述，这是上下文缩减的核心：把工具结果卸载到文件系统，生成 compact 版本，用于从智能体上下文窗口修剪陈旧 token。与 Claude Code 类似，Manus 使用基础工具（如 glob 和 grep）检索文件系统，而不需要索引（如 vectorstores）。

模型选择

Manus 不会绑定单一模型，而是按任务路由：编码可能用 Claude，多模态任务可能用 Gemini，数学与推理可能用 OpenAI。总体上，Manus 的模型选择受成本驱动，而 KV cache 效率是核心因素。

Manus 使用缓存（例如系统指令、旧工具结果等）以降低多轮智能体执行中的成本与时延。Peak 提到，开源模型上的分布式 KV cache 基础设施实现难度较高，而头部模型提供商支持更完善。对于某些（智能体）场景，这种缓存支持会让头部模型在实践中更便宜。

带着 Bitter Lesson 思维来构建

我们最后讨论了 Bitter Lesson。我一直很关注它对 AI 工程的启示。Claude Code 的作者 Boris Cherny 提到，Bitter Lesson 影响了他让 Claude Code 保持“少主观预设”的决定，这样更容易适配模型改进。

在持续进化的模型之上构建，意味着必须接受持续变化。Peak 提到 Manus 自 3 月上线以来已经重构了五次！

此外，Peak 警告说，随着模型进步，智能体 harness 可能反过来限制性能；这正是 Bitter Lesson 指出的挑战。我们在某个时间点加入结构以提升性能，但随着算力（模型）增长，这些结构可能会变成上限。

为防止这种情况，Peak 建议在不同模型强度下运行智能体评测。如果更强模型并未带来性能提升，你的 harness 可能正在拖后腿。这可以帮助判断你的 harness 是否“面向未来”。

Hyung Won Chung（OpenAI/MSL）关于该主题的演讲进一步强调：随着模型改进，需要持续重新评估结构（例如你的 harness/假设）。

在当前算力和数据水平下，添加必要结构。之后要移除它们，因为这些捷径会成为后续提升的瓶颈。

结论

让智能体访问计算机（如文件系统、终端、工具集）是我们在许多智能体（包括 Manus）中看到的常见模式。它支持几类上下文工程策略：

1. 卸载上下文

将工具结果外置存储：把完整工具结果保存到文件系统（而非上下文），并按需通过 glob 和 grep 等工具访问

将动作下推到沙箱：使用少量函数调用（Bash、文件系统访问）在沙箱中执行大量工具，而不是把每个工具都绑定为 tool

2. 缩减上下文

压缩陈旧结果：当上下文变满时，用引用（如文件路径）替换旧工具结果；保留最近结果的完整内容以引导下一次决策

按需摘要：当压缩收益递减后，对完整轨迹应用基于 schema 的摘要

3. 隔离上下文

将离散任务交给子智能体：把任务分配给各自拥有上下文窗口的子智能体，主要目的是隔离上下文（而非按角色分工）

有意识地共享上下文：简单任务仅传指令；复杂任务（例如子智能体需要更多上下文）则传完整上下文（如轨迹与共享文件系统）

最后还要注意，随着模型改进，要确保你的 harness 不会限制性能（即“Bitter Lesson-pilled”）。通过不同模型强度测试来验证这一点。简单、少主观预设的设计通常更容易适配模型进步。最后，不要害怕随着模型进步重建你的智能体（Manus 自 3 月以来已重构 5 次）！

面向 AI 智能体的上下文工程：来自构建 Manus 的经验

2025/7/18 --Yichao 'Peak' Ji

在项目最初阶段，我和团队面临一个关键决策：我们应该基于开源基础模型训练端到端智能体模型，还是构建在头部模型能力之上的智能体？

在我做 NLP 的第一个十年里，我们根本没有这种选择。很早以前的  时代（没错，已经过去七年了），模型必须先微调并评估，才能迁移到新任务。即使当时模型规模远小于今天的 LLM，这个过程每次迭代也常常需要数周。对于快速变化的应用，尤其是 PMF 前阶段，如此缓慢的反馈回路是不可接受的。这是我上一家创业公司留下的惨痛教训：当时我从零训练模型用于  和语义搜索。后来  和  出现，我的自研模型一夜之间失去意义。讽刺的是，正是这些模型开启了 in-context learning，也打开了一条全新路径。

这份来之不易的经验让选择变得清晰：Manus 押注上下文工程。这样我们可以把改进周期从“几周”缩短到“几小时”，并让产品与底层模型保持正交：如果模型进步是上涨的潮水，我们希望 Manus 是船，而不是固定在海底的桩。

但上下文工程远非直线式推进。它是一门实验科学。我们已经四次重建智能体框架，每次都是在发现了更好的上下文塑形方式之后。我们把这种通过架构试探、提示词微调和经验性猜测完成的手工过程，戏称为“随机研究生下降（Stochastic Graduate Descent）”。它并不优雅，但很有效。

这篇文章分享我们通过“SGD”找到的局部最优。如果你正在构建自己的 AI 智能体，希望这些原则能帮助你更快收敛。

围绕 KV-Cache 设计

如果只能选一个指标，我会说 KV-cache 命中率是生产阶段 AI 智能体最重要的单一指标。它直接影响时延和成本。要理解原因，先看  是如何运行的：

在接收用户输入后，智能体会通过一连串工具调用来完成任务。每次迭代中，模型根据当前上下文从预定义动作空间选择动作。该动作随后在环境中执行（例如 Manus 的虚拟机沙箱），产生 observation。动作和 observation 会追加到上下文，形成下一轮迭代输入。循环持续到任务完成。

可以想见，上下文会随每一步增长，而输出（通常是结构化函数调用）相对较短。与聊天机器人相比，这让智能体的 prefilling 与 decoding 比例极度失衡。例如在 Manus 中，平均输入输出 token 比约为 100:1。

幸运的是，具有相同前缀的上下文可以利用 ，显著降低首 token 时间（TTFT）和推理成本，无论你是自托管模型还是调用推理 API。这不是小幅节省：以 Claude Sonnet 为例，缓存输入 token 成本为 0.30 USD/MTok，而未缓存是 3 USD/MTok，差了 10 倍。

从上下文工程角度看，提高 KV-cache 命中率涉及几个关键实践：

保持提示词前缀稳定。由于 LLM 的  特性，即使只差一个 token，也会让该 token 之后的缓存失效。常见错误是在系统提示开头加入时间戳，尤其精确到秒。它确实能让模型告诉你当前时间，但也会毁掉缓存命中率。

让上下文保持 append-only。避免修改之前的动作或 observation。确保序列化是确定性的。许多编程语言和库在序列化 JSON 对象时不保证稳定的键顺序，这会悄悄破坏缓存。

需要时显式标记缓存断点。一些模型提供商或推理框架不支持自动增量前缀缓存，而需要在上下文中手动插入缓存断点。设置断点时要考虑缓存过期风险，至少应保证断点覆盖系统提示末尾。

此外，如果你用  这类框架自托管模型，请确保启用了 ，并使用 session ID 等技术在分布式 worker 间做一致路由。

遮蔽，而不是删除（Mask, Don't Remove）

随着智能体能力扩展，其动作空间自然会变复杂，直白说就是工具数量爆炸。近期  的流行更是火上浇油。如果你允许用户自定义工具，相信我：一定会有人把数百个神秘工具塞进你精心设计的动作空间。结果是模型更容易选错动作或走低效路径。简言之，武装到牙齿的智能体反而更“笨”了。

一个自然反应是设计动态动作空间，比如用类似  的方式按需加载工具。我们在 Manus 里也试过。但实验给出一条明确规则：除非绝对必要，不要在迭代中途动态增删工具。原因有两点：

在大多数 LLM 中，工具定义经序列化后位于上下文前部，通常在系统提示前后。因此任何改动都会让后续所有动作与 observation 的 KV-cache 失效。

当历史动作和 observation 仍引用当前上下文中已不再定义的工具时，模型会困惑。缺少  时，这通常会导致 schema 违例或动作幻觉。

为在提升动作选择的同时解决这些问题，Manus 使用上下文感知的  管理工具可用性。它不是删除工具，而是在解码时对 token logits 做 masking，根据当前上下文阻止（或强制）某些动作被选中。

在实践中，大多数模型提供商和推理框架都支持某种 response prefill，可以在不修改工具定义的前提下约束动作空间。函数调用通常有三种模式（以下以 NousResearch 的  为例）：

Auto – 模型可以选择调用或不调用函数。实现方式是仅预填充回复前缀：<|im_start|>assistant

Required – 模型必须调用函数，但函数选择不受限。实现方式是预填充到 tool call token：<|im_start|>assistant<tool_call>

Specified – 模型必须从特定子集中调用函数。实现方式是预填充到函数名开头：<|im_start|>assistant<tool_call>{"name": “browser_

借助这个机制，我们可以通过直接遮蔽 token logits 来约束动作选择。例如，当用户给出新输入时，Manus 必须立即回复而不是先执行动作。我们还故意把动作名设计为统一前缀，如所有浏览器相关工具都以 browser_ 开头，命令行工具以 shell_ 开头。这样无需 stateful logits processor，也能在特定状态下轻松限制智能体只能从某组工具中选择。

这些设计有助于确保 Manus 的智能体循环在模型驱动架构下仍保持稳定。

把文件系统当作上下文（Use the File System as Context）

现代头部 LLM 已提供 128K 甚至更大的上下文窗口。但在真实智能体场景里，这往往不够，有时甚至是负担。常见痛点有三类：

Observation 可能非常大，尤其当智能体处理网页或 PDF 这类非结构化数据时，很容易突破上下文上限。

即便窗口技术上支持，模型性能也往往会在某个上下文长度后下降。

即使有前缀缓存，长输入仍然昂贵。你依然要为传输和预填充每个 token 付费。

为应对这些问题，许多智能体系统会实现上下文截断或压缩策略。但压缩过激必然导致信息丢失。问题是根本性的：智能体本质上需要基于全部先前状态预测下一动作，而你无法可靠预判哪个 observation 会在十步后变关键。从逻辑上讲，任何不可逆压缩都带风险。

这就是为什么在 Manus 中我们把文件系统视为终极上下文：容量无限、天然持久、且智能体本身可直接操作。模型学会按需读写文件，把文件系统不仅当存储，还当结构化、外化的记忆。

我们的压缩策略始终可恢复。例如，只要保留 URL，就可以把网页内容从上下文中移除；只要沙箱中保留文档路径，就可以省略文档内容。这样 Manus 可缩短上下文长度而不永久丢失信息。

开发这个功能时，我一直在想：要让状态空间模型（SSM）在智能体场景里高效工作需要什么。与 Transformer 不同，SSM 缺乏完整注意力机制，难以处理长程反向依赖。但如果它们能掌握基于文件的记忆，把长期状态外化而不是保留在上下文里，那么它们的速度与效率或许会开启一类新智能体。Agentic SSM 可能才是  的真正继任者。

通过复述来操纵注意力（Manipulate Attention Through Recitation）

如果你用过 Manus，可能注意到一个现象：处理复杂任务时，它倾向于创建 todo.md，并随着任务推进逐步更新、勾选完成项。

这不只是“可爱行为”，而是一个刻意设计的注意力操控机制。

一个典型 Manus 任务平均需要约 50 次工具调用。这是很长的循环；而 Manus 依赖 LLM 做决策，因此在长上下文或复杂任务中容易跑题或忘记早期目标。

通过不断重写 todo 列表，Manus 等于把目标持续复述到上下文末端。这会把全局计划推入模型的近期注意力范围，避免“lost-in-the-middle”问题并减少目标偏移。本质上，它在不改变架构的前提下，用自然语言偏置自身注意力到任务目标上。

把错误信息保留下来（Keep the Wrong Stuff In）

智能体会犯错。这不是 bug，而是现实。语言模型会幻觉、环境会报错、外部工具会异常、各种边缘情况会出现。在多步任务里，失败不是例外，而是循环的一部分。

然而常见冲动是掩盖这些错误：清理轨迹、重试动作，或重置模型状态并交给神奇的 ""。这看起来更安全、更可控，但代价是：擦除失败就擦除了证据；没有证据，模型就无法适应。

按我们的经验，提升智能体行为最有效的方法之一其实很简单：把走错的路径留在上下文里。当模型看到失败动作及其 observation 或堆栈信息时，它会隐式更新内部信念，把先验从类似动作上移开，降低重复犯错概率。我们甚至认为，错误恢复能力是“真正智能体行为”的最清晰指标之一。但在学术工作和公开基准里，它仍被低估，后者常在理想条件下只关注任务成功率。

别被自己 few-shot 了（Don't Get Few-Shotted）

 是提升 LLM 输出的常见技术。但在智能体系统中，它可能以微妙方式反噬。

语言模型非常擅长模仿，会复制上下文中的行为模式。如果上下文里充满相似的历史动作-观察对，模型就会沿用该模式，即便它已经不是最优选择。

这在涉及重复决策或动作的任务中很危险。比如用 Manus 批量评审 20 份简历时，智能体经常进入节奏化重复，仅仅因为上下文里就是那样。这会导致漂移、过度泛化，有时甚至产生幻觉。

解决方法是提高多样性。Manus 在动作与 observation 中注入少量结构化变化，比如不同序列化模板、替代表述、顺序或格式上的轻微噪声。这种受控随机性能打破模式并微调模型注意力。换言之，不要把自己 few-shot 进死胡同。上下文越一致，智能体越脆弱。

结论

上下文工程仍是一门新兴科学，但对智能体系统来说已经是必需项。模型也许越来越强、越来越快、越来越便宜，但原始能力再强，也替代不了记忆、环境和反馈。你如何塑造上下文，最终定义了智能体如何表现：运行多快、恢复多好、扩展多远。

在 Manus，我们通过反复重写、撞墙和面向数百万用户的真实测试学到了这些经验。这里分享的并非普适真理，但确实是对我们有效的模式。如果它们能帮你少经历哪怕一次痛苦迭代，这篇文章就达到了目的。

智能体未来将由一个个上下文构建起来。把它们工程化好。

Wide Research：超越上下文窗口

AI 驱动研究的承诺一直很诱人：把信息收集与综合这些繁琐工作交给智能系统，让人类认知专注于更高层级的分析与决策。然而，任何在非平凡场景中深入使用这些系统的人都会遇到一个令人沮丧的现实：在多主题研究任务做到第八或第九项时，AI 开始“编造”。

不是简单化。不是更简洁地总结。是编造。

这不是提示词工程问题，也不是模型能力问题。这是一个架构约束，自这些 AI 研究工具诞生以来就一直悄悄限制着它们的效用。而 Wide Research 正是为突破这一约束而设计。

上下文窗口：一个基础瓶颈

每个大语言模型都在上下文窗口内运行。这是一个有限记忆缓冲区，限制了模型在任意时刻可主动处理的信息量。现代模型把这个边界推高了很多：从 4K token 到 32K、128K，最近甚至到 1M token。

但问题依旧存在。

当你要求 AI 研究多个实体（比如 50 家公司、30 篇论文或 20 个竞品）时，上下文窗口会迅速被填满。占用它的不仅是每个实体的原始信息，还有：

原始任务说明与要求

用于保证输出格式一致的结构模板

每个条目的中间推理与分析

交叉引用与比较笔记

所有前序条目的累积上下文

当模型做到第八或第九项时，上下文窗口已承受巨大压力。模型面临一个几乎不可能的选择：显式失败，或开始偷工减料。

它总会选后者。

编造阈值（The Fabrication Threshold）

在实际中会发生什么：

第 1-5 项：模型执行真实研究。它会检索信息、交叉核验来源，并输出详细且准确的分析。

第 6-8 项：质量开始微妙下降。描述变得更泛化。模型开始更多依赖先前模式，而不是新检索。

第 9+ 项：模型进入编造模式。由于在上下文溢出压力下无法维持全面研究负荷，它会根据统计模式生成“看起来可信”的内容，而非真实调查结果。

这些编造通常很“高级”：语气权威、格式完全一致、语法流畅、风格也与前面真实条目保持一致。

但它们往往是错的。

竞品分析可能给不具备该能力的公司硬塞功能。文献综述可能引用带有虚构结论的论文。产品对比可能捏造价格档位或参数。

最阴险之处在于：这些编造如果不做人工核验很难发现，而这又直接抵消了自动化研究的意义。

为什么更大的上下文窗口无法解决

直觉上的反应是继续扩大上下文窗口。32K 不够就上 128K，仍不够就推到 200K 以上。

这种做法误解了问题本质。

第一，上下文衰减不是二元的。模型不会在整个上下文窗口中保持完美召回。研究显示，检索准确率会随与当前位置距离增加而下降，这就是“lost in the middle”现象。窗口开头和结尾的信息比中间信息更容易被召回。

第二，处理成本增长并不线性。处理 400K token 上下文的成本不只是 200K 的两倍，而是在时间和算力上呈指数级恶化。对很多场景来说，这在经济上不可行。

第三，问题是认知负荷。即使有无限上下文，让单个模型在几十个独立研究任务上保持一致质量仍会形成认知瓶颈。模型必须在条目间频繁切换、维护比较框架、保持文风一致，同时还要完成核心研究任务。

第四，上下文长度压力。模型的“耐心”某种程度由训练数据中样本长度分布决定。但当前语言模型的后训练数据混合仍以较短轨迹为主，偏向聊天交互。因此，当助手消息长度超过某个阈值时，模型会自然出现一种上下文长度压力，促使其更快进入总结，或退化为不完整表达形式（如项目符号）。

上下文窗口确实是约束。但它只是更深层架构限制的表象：单处理器、串行范式。

架构转变：并行处理

Wide Research 代表了 AI 系统处理大规模研究任务方式的一次根本重构。它不再让一个处理器串行处理 n 个条目，而是部署 n 个并行子智能体同时处理 n 个条目。

Wide Research 架构

当你发起 Wide Research 任务时，系统按如下方式运行：

1. 智能拆解

主控制器会分析你的请求，并将其拆解为可并行、相互独立的子任务。这一步包含任务结构理解、依赖识别与连贯子规格生成。

2. 子智能体委派

对每个子任务，系统都会启动一个专用子智能体。关键是，它们不是轻量进程，而是完整 Manus 实例，每个实例都有：

完整虚拟机环境

完整工具库访问权限（搜索、浏览、代码执行、文件处理）

独立互联网连接

全新、空白的上下文窗口

3. 并行执行

所有子智能体同时运行。每个子智能体只聚焦自己的条目，并以处理单条任务时相同的研究深度执行分析。

4. 集中协调

主控制器保持全局监督，在子智能体完成后收集结果。重要的是，子智能体彼此不通信，所有协调都通过主控制器完成。这可避免上下文污染并保持独立性。

5. 综合与整合

所有子智能体回传后，主控制器将结果综合为单一、连贯且全面的报告。由于主控制器不承担原始研究负担，这一步可充分利用其上下文容量。

为什么这改变了一切

规模化下的一致质量

每个条目都得到同等处理。第 50 项会和第 1 项一样被充分研究。没有质量衰减曲线、没有编造阈值、没有断崖式下滑。

真正的横向扩展

要分析 10 项？系统部署 10 个子智能体。要分析 500 项？部署 500 个。该架构随任务规模线性扩展，而不是像基于上下文的方法那样指数恶化。

显著加速

由于子智能体并行运行，分析 50 项的真实耗时大致接近分析 5 项。瓶颈从串行处理时间转移到综合时间，而综合只占整体任务较小部分。

更低幻觉率

每个子智能体都在自己的“认知舒适区”运行。新鲜上下文 + 单一聚焦任务，不会有编造压力。子智能体可以执行真实研究、核验事实并维持准确性。

独立性与可靠性

由于子智能体不共享上下文，一个子智能体中的错误或幻觉不会传播给其他子智能体。每份分析都相对独立，从而降低系统性风险。

超越单处理器范式

Wide Research 不只是一个功能，它代表了从单处理器范式向编排式并行架构的根本迁移。AI 系统的未来不在于无限增大上下文窗口，而在于智能任务拆解与并行执行。

我们正从“AI 助手”时代，走向“AI 劳动力”时代。

何时使用 Wide Research：任何涉及多个相似条目且要求一致分析的任务，例如竞品研究、文献综述、批量处理、多资产生成。

何时不该使用：高度串行、每一步都强依赖前一步的任务，或小任务（少于 10 项）中单处理器更具成本效益的场景。

我们如何构建多智能体研究系统

发表于 2025 年 6 月 13 日

我们的 Research 功能使用多个 Claude 智能体，更有效地探索复杂主题。我们分享了构建这个系统时遇到的工程挑战和经验教训。

Claude 现在具备 Research 能力，可跨网页、Google Workspace 以及各类集成进行搜索以完成复杂任务。

这个多智能体系统从原型到生产的过程，让我们在系统架构、工具设计和提示词工程方面学到了关键经验。多智能体系统由多个智能体（在循环中自主使用工具的 LLM）协同组成。我们的 Research 功能包含一个智能体：它基于用户查询规划研究流程，然后使用工具创建并行智能体同时搜索信息。多智能体系统会引入智能体协调、评估和可靠性的新挑战。

这篇文章拆解了对我们有效的原则，希望你在构建自己的多智能体系统时也能借鉴。

多智能体系统的优势

研究工作面对的是开放问题，所需步骤很难提前准确预测。探索复杂主题时，你无法硬编码一条固定路径，因为过程天然是动态且路径依赖的。人类做研究时，也会根据新发现持续调整方法，沿着调查中出现的新线索推进。

这种不确定性使 AI 智能体特别适合研究任务。研究要求系统在调查展开时具备转向或探索旁支关联的灵活性。模型必须在多轮中自主运行，根据中间发现决定下一步方向。线性、一次性的流水线无法胜任这类任务。

搜索的本质是压缩：从海量语料中提炼洞见。子智能体通过并行运行并使用各自上下文窗口探索问题不同方面，然后把最重要 token 压缩给主研究智能体，从而促进这种压缩。每个子智能体还带来关注点分离：不同工具、不同提示、不同探索轨迹，减少路径依赖并支持深入且独立的调查。

当智能达到某个阈值后，多智能体系统会成为扩展性能的关键方式。例如，尽管过去 10 万年单个人类更聪明了，但由于群体智能和协作能力，信息时代的人类社会能力指数级提升。即便是通用智能体，单体运行也有限制；智能体群体可以完成更多事情。

我们的内部评估显示，多智能体研究系统尤其擅长广度优先查询，即需要同时沿多个独立方向推进的任务。我们发现，以 Claude Opus 4 为主智能体、Claude Sonnet 4 为子智能体的多智能体系统，在内部研究评测上比单智能体 Claude Opus 4 高 90.2%。例如，当要求找出标普 500 信息技术公司的全部董事会成员时，多智能体系统通过任务拆解给子智能体得出了正确答案，而单智能体系统在缓慢串行搜索中失败。

多智能体系统之所以有效，主要因为它们能“花够 token”来解决问题。在我们的分析中，BrowseComp 评测（测试浏览智能体定位难找信息的能力）中 95% 的性能方差可由三个因素解释。仅 token 使用量就解释了 80% 的方差，另外两个因素是工具调用次数和模型选择。这个发现验证了我们的架构：把工作分发到拥有独立上下文窗口的智能体上，从而增加并行推理容量。最新 Claude 模型会显著提升 token 使用效率：升级到 Claude Sonnet 4 带来的性能增益，甚至大于在 Claude Sonnet 3.7 上把 token 预算翻倍。多智能体架构能有效扩展超出单智能体极限的任务 token 用量。

缺点也很明显：在实践中，这类架构会非常快地消耗 token。我们的数据里，智能体通常比聊天交互多用约 4× token，而多智能体系统约为聊天的 15×。要具备经济可行性，多智能体系统需要任务本身有足够高价值，能覆盖这部分额外成本。此外，一些要求所有智能体共享同一上下文，或智能体间依赖关系很多的领域，目前并不适合多智能体系统。比如大多数编码任务中，真正可并行化的子任务比研究少得多，而且 LLM 智能体在实时协调和委派上还不够好。我们的经验是，多智能体系统最擅长高价值任务：重并行、信息量超过单一上下文窗口、并需要对接众多复杂工具。

Research 的架构总览

我们的 Research 系统采用多智能体架构，使用 orchestrator-worker 模式：主智能体负责协调流程，同时把工作委派给并行运行的专业子智能体。

多智能体架构运行示意：用户查询流入主智能体，主智能体创建专业子智能体并行搜索不同方面。

当用户提交查询时，主智能体会先分析、制定策略，再派生子智能体并行探索不同方面。正如上图所示，子智能体充当智能过滤器：迭代使用搜索工具收集信息（此例为 2025 年 AI 智能体公司），然后把公司列表返回给主智能体，由其汇总最终答案。

传统基于检索增强生成（RAG）的方法使用静态检索。也就是说，它们会抓取一组与输入查询最相似的切片，并据此生成响应。相比之下，我们的架构使用多步搜索：动态发现相关信息、适应新发现，并分析结果以形成高质量答案。

流程图展示了多智能体 Research 系统的完整工作流。当用户提交查询时，系统会创建一个 LeadResearcher 智能体并进入迭代研究流程。LeadResearcher 首先会思考研究方法并把计划保存到 Memory 以持久化上下文，因为上下文窗口一旦超过 200,000 tokens 就会被截断，而保留计划至关重要。随后它会创建专门的 Subagents（这里展示了两个，实际可为任意数量）并分配具体研究任务。每个 Subagent 独立执行网页搜索，结合交错思考评估工具结果，再将发现返回给 LeadResearcher。LeadResearcher 进行综合，并判断是否需要更多研究；若需要，它可以继续创建子智能体或调整策略。收集到足够信息后，系统退出研究循环，并将全部发现交给 CitationAgent。CitationAgent 会处理文档和研究报告，定位具体引用位置，确保所有结论都可追溯到来源。最终，带引用的研究结果返回给用户。

研究智能体的提示词工程与评估

多智能体系统与单智能体系统有关键差异，其中之一是协调复杂度会迅速上升。早期智能体会出现这类错误：简单查询也生成 50 个子智能体、为不存在来源无休止地全网搜索、因过多更新彼此干扰。由于每个智能体都由提示词驱动，提示词工程成了我们改进行为的主要杠杆。下面是我们在提示智能体方面学到的一些原则：

像你的智能体那样思考。要迭代提示词，就必须理解其影响。为此我们用 Console 搭建了与生产系统完全一致提示词与工具的模拟环境，逐步观察智能体行为。失败模式会立刻暴露：明明结果已足够却继续搜索、搜索词过度冗长、选择错误工具。有效提示依赖于对智能体建立准确心智模型，这会让最有价值的改动变得显而易见。

教会编排器如何委派。在我们的系统里，主智能体把查询拆解为子任务并描述给子智能体。每个子智能体都需要目标、输出格式、工具和来源使用指引，以及清晰任务边界。若任务描述不够细，智能体会重复劳动、遗漏信息，或找不到必要内容。我们起初允许主智能体给出简短指令，比如“研究芯片短缺”，但很快发现这类指令常常过于模糊，导致子智能体误解任务，或与其他子智能体做完全相同搜索。例如，一个子智能体研究 2021 年汽车芯片危机，另外两个却重复调研 2025 年当前供应链，分工完全无效。

按查询复杂度匹配投入。智能体不擅长自己判断不同任务该投入多少资源，所以我们把扩展规则写进提示词。简单事实查询只需 1 个智能体与 3-10 次工具调用；直接对比可能需要 2-4 个子智能体、每个 10-15 次调用；复杂研究可用 10+ 子智能体并明确责任划分。这些显式准则帮助主智能体高效分配资源，也避免了在简单查询上过度投入（这是早期常见失败模式）。

工具设计与选择至关重要。智能体-工具接口之于智能体，正如人机交互之于人。用对工具很高效，很多时候甚至是刚需。比如，若信息只存在于 Slack，而智能体却去网上搜，那从起点就错了。有了 MCP 服务器给模型接入外部工具后，问题更复杂，因为智能体会遇到大量从未见过且描述质量参差不齐的工具。我们给智能体加入显式启发式：先查看所有可用工具、根据用户意图匹配工具、外部广域探索用 web 搜索、优先专用工具而非通用工具。糟糕的工具描述会把智能体带偏，所以每个工具都必须有明确用途和清晰描述。

让智能体改进自己。我们发现 Claude 4 模型本身就是优秀的提示词工程师。给它一个提示词和失败模式，它可以诊断失败原因并提出改进建议。我们甚至做了一个工具测试智能体：给它有缺陷的 MCP 工具后，它会尝试使用，再重写工具描述来规避失败。通过几十次测试，这个智能体发现了关键细节和 bug。改进工具易用性的这一流程，让后续智能体在新描述下的任务完成时间降低了 40%，因为它们能避免大多数错误。

先广后深。搜索策略应模仿专家研究者：先扫全局，再钻细节。智能体常默认用过长、过具体查询，结果返回很少。我们通过提示词纠正这一倾向：先用短而宽的查询，评估可用信息，再逐步收窄焦点。

引导思考过程。扩展思考模式会让 Claude 输出额外 token 形成可见思考过程，可作为可控 scratchpad。主智能体用思考来规划方法：判断哪些工具适配任务、确定查询复杂度和子智能体数量、定义每个子智能体角色。测试显示，扩展思考提升了指令遵循、推理与效率。子智能体也会先规划，再在工具结果后做交错思考以评估质量、识别缺口并改进下一查询，使其能更好适应各种任务。

并行工具调用显著改变速度与性能。复杂研究任务天然需要探索大量来源。我们早期智能体串行搜索，速度极慢。为提速我们引入两种并行：(1) 主智能体并行启动 3-5 个子智能体，而非串行；(2) 子智能体并行使用 3+ 工具。这些改动让复杂查询研究时间最多缩短 90%，使 Research 能在几分钟内完成过去要数小时的工作，并覆盖比其他系统更多信息。

我们的提示策略重点是灌输良好启发式，而非僵化规则。我们研究了熟练人类如何做研究，并把这些策略编码进提示词：如把难题拆成小任务、谨慎评估来源质量、基于新信息调整搜索方法、识别何时要做深度（深入一个主题）与广度（并行探索多个主题）权衡。我们也通过显式护栏主动缓解副作用，防止智能体失控。最后，我们重点建设了带可观测性与测试样例的快速迭代闭环。

有效评估智能体

高质量评估是构建可靠 AI 应用的基础，智能体也不例外。但多智能体评估有独特挑战。传统评估常假设 AI 每次走同一路径：输入 X 应走路径 Y 产出 Z。多智能体并非如此。即使起点相同，智能体也可能走完全不同但同样有效的路径达成目标。一个智能体可能搜 3 个来源，另一个搜 10 个；它们还可能用不同工具得到同一答案。因为我们并不总知道“正确步骤”是什么，通常也无法仅检查它们是否遵循了预设步骤。我们需要更灵活的评估：既判断结果是否正确，也判断过程是否合理。

立刻用小样本开始评估。早期智能体开发阶段，改动影响通常很大，因为有大量低垂果实。一次提示词调整可能把成功率从 30% 拉到 80%。在这种大效应下，只需少量样例就能看出变化。我们从约 20 个代表真实使用模式的查询开始，通常已足以看清改动效果。我们经常听到团队因为认为“只有几百样例的大评测才有意义”而推迟做 eval。实际上，更好的做法是立刻用少量样例启动小规模测试，而不是等到能做完备评测再开始。

做得好的 LLM-as-judge 可规模化。研究输出很难用程序评估，因为它们是自由文本，且很少只有一个正确答案。LLM 非常适合做评分器。我们使用 LLM judge，按评分标准评估输出：事实准确性（结论是否匹配来源）、引用准确性（引用来源是否支持结论）、完整性（是否覆盖所有要求方面）、来源质量（是否优先用一手来源而非低质量二手来源）、工具效率（是否在合理次数内使用了正确工具）。我们尝试过多个评审分别打分，但发现“单次 LLM 调用 + 单一提示词 + 0.0-1.0 分数和通过/失败结果”最稳定，也最贴近人工判断。该方法在评测题有明确答案时尤其有效，此时可让 LLM 仅判断答案正确性（例如是否准确列出研发预算前三的制药公司）。使用 LLM 评审使我们可规模化评估数百个输出。

人工评估能发现自动化遗漏。人工测试者能发现 eval 漏掉的边缘情况，如非常规查询下的幻觉回答、系统故障或细微来源偏差。以我们为例，人工测试发现早期智能体会系统性偏向 SEO 内容农场，而不是权威但排名没那么高的来源（如学术 PDF 或个人博客）。在提示词中加入来源质量启发式后，该问题得到缓解。即便在自动化评估时代，人工测试仍不可替代。

多智能体系统会出现未被显式编程的涌现行为。例如，主智能体的小改动可能不可预测地改变子智能体行为。要成功，需要理解交互模式，而不只是单体行为。因此，这类智能体的最佳提示词不应只是严格指令，而应是协作框架：定义分工、解题方法和投入预算。把这件事做好依赖于精细提示词与工具设计、稳健启发式、可观测性以及紧密反馈回路。你可以查看我们 Cookbook 里的开源提示词示例。

生产可靠性与工程挑战

在传统软件里，一个 bug 可能破坏功能、降低性能或引发故障。在智能体系统里，小改动会级联成大行为变化，这使得编写需要在长时间运行中维护状态的复杂智能体代码变得格外困难。

智能体是有状态的，错误会复合。智能体可长时间运行并跨许多工具调用维持状态。这意味着我们需要可持久执行代码，并在过程中处理错误。若缺乏有效缓解，微小系统故障对智能体都可能是灾难性的。错误发生时，我们不能简单从头重启：重启成本高且用户体验差。我们因此构建了可从出错点恢复的系统。我们也利用模型智能优雅处理问题：例如显式告诉智能体某工具失效并让其自适应，效果出乎意料地好。我们把基于 Claude 的智能体适应性与确定性保障（重试逻辑、定期 checkpoint）结合使用。

调试需要新方法。智能体会做动态决策，在不同运行间是非确定性的，即使提示词相同。这让调试更难。例如用户会反馈智能体“找不到明显信息”，但我们看不出原因：是查询词不佳？来源选择差？还是工具故障？加入完整生产追踪后，我们才能诊断失败原因并系统修复。除标准可观测性外，我们还监控智能体决策模式和交互结构，同时不监控单个对话内容以保护隐私。这种高层可观测性帮助我们定位根因、发现意外行为并修复常见故障。

部署需要精细协同。智能体系统是由提示词、工具与执行逻辑组成的高状态网络，并且几乎持续运行。这意味着每次部署更新时，智能体可能处在流程任意位置。我们必须避免“善意改动”破坏正在运行的智能体。我们不能让所有智能体同时切到新版本。相反，我们使用彩虹部署（rainbow deployments），通过在新旧版本同时运行的前提下逐步切流，避免中断在途智能体。

同步执行会形成瓶颈。当前主智能体同步执行子智能体：必须等待一批子智能体完成后再继续。这简化了协调，但在智能体间信息流上形成瓶颈。例如主智能体无法实时引导子智能体，子智能体之间也无法协同，系统还可能被单个慢子智能体阻塞。异步执行能带来更多并行：智能体并发工作，并在需要时创建新子智能体。但异步也带来结果协调、状态一致性和错误传播挑战。随着模型能处理更长更复杂研究任务，我们预计其性能收益将值得这份复杂度。

结论

构建 AI 智能体时，最后一公里往往占据了大部分旅程。在开发机上可运行的代码库，要成为可靠生产系统往往还需大量工程化投入。智能体系统中错误的复合特性意味着：传统软件里的小问题，足以让智能体彻底跑偏。一步失败可能让智能体走上完全不同轨迹，导致不可预测结果。正因本文所述原因，原型到生产的鸿沟往往比预期更宽。

尽管如此，多智能体系统在开放式研究任务中已证明其价值。用户反馈称，Claude 帮他们发现了此前没想到的商业机会、梳理了复杂医疗方案、解决了棘手技术 bug，并通过发现独自难以找到的研究关联，节省了多达数天工作量。只要有审慎工程、全面测试、注重细节的提示词与工具设计、稳健运维实践，以及对当前智能体能力有深刻理解的研究/产品/工程团队紧密协作，多智能体研究系统就能可靠地规模化运行。我们已经看到这些系统正在改变人们解决复杂问题的方式。

附录

下面是一些关于多智能体系统的补充建议。

针对跨多轮修改状态的智能体做终态评估。评估会在多轮对话中修改持久状态的智能体，有其独特挑战。与只读研究任务不同，每个动作都可能改变后续步骤环境，形成传统评估方法难以处理的依赖。我们发现，聚焦终态评估比逐轮分析更有效。不要判断智能体是否遵循了某条固定流程，而应判断它是否达到了正确最终状态。这种方法承认智能体可以通过不同路径到达同一目标，同时保证结果符合预期。对于复杂工作流，可把评估拆为离散检查点，检查关键状态变化是否发生，而不是尝试验证每个中间步骤。

长时对话管理。生产智能体常进行跨越数百轮的对话，需要谨慎的上下文管理策略。对话变长后，标准上下文窗口会不够用，需要智能压缩和记忆机制。我们采用了这样的模式：智能体会在进入新任务前先总结已完成工作阶段，并把关键信息存入外部记忆。当接近上下文上限时，智能体可派生上下文干净的新子智能体，同时通过精细交接维持连续性。此外，智能体可从记忆中取回先前上下文（例如研究计划），而不是到达上限时丢失历史工作。这种分布式方式既防止上下文溢出，也保留了长交互的一致性。

将子智能体输出写入文件系统，以最小化“传话游戏”。某些结果类型下，子智能体输出可以绕过主协调者，从而提升保真度和性能。与其要求子智能体把所有内容都经由主智能体转述，不如实现 artifact 系统，让专用智能体产生可独立持久化的输出。子智能体通过工具把工作存到外部系统，再把轻量引用返回给协调者。这样可避免多阶段处理中信息损失，并减少把大输出复制进对话历史带来的 token 开销。该模式尤其适合结构化输出（代码、报告、数据可视化），因为子智能体的专用提示往往比经过通用协调者过滤后效果更好。

什么是工具？

在计算领域，确定性系统在相同输入下每次产生相同输出，而非确定性系统（如智能体）即使起始条件相同也可能给出不同响应。

传统软件开发中，我们建立的是确定性系统之间的契约。例如 `getWeather("NYC")` 这样的函数调用，每次都会以同样方式获取纽约天气。

工具是一种新的软件形态，体现的是确定性系统与非确定性智能体之间的契约。当用户问“今天要带伞吗？”时，智能体可能调用天气工具，可能凭常识回答，也可能先追问位置。偶尔智能体也会幻觉，或没弄明白如何使用工具。

这意味着为智能体写软件时需要根本性转变：不能再按给开发者或其他系统写函数/API 的方式写工具和 MCP 服务器，而要为智能体而设计。

我们的目标是通过工具扩大智能体可有效发挥作用的“表面积”，让它们能采用多样且成功的策略解决广泛任务。幸运的是，我们的经验是：对智能体最“易用”的工具，往往对人类也同样直观。

如何编写工具

本节我们介绍如何与智能体协作来编写并改进工具。先快速搭建工具原型并在本地测试，再运行全面评估衡量后续改动。你可以与智能体并肩迭代“评估-改进”循环，直到智能体在真实任务上达到强表现。

构建原型

在没有亲手实践前，很难预判哪些工具对智能体友好，哪些不友好。先快速搭建原型。如果你用 Claude Code 来写工具（甚至一轮成型），建议把工具所依赖的软件库、API 或 SDK 文档（包括可能的 MCP SDK）提供给 Claude。面向 LLM 的文档常以扁平 `llms.txt` 形式出现在官方文档站点（我们的 API 就有）。

把工具封装到本地 MCP 服务器或 Desktop 扩展（DXT）中，可让你在 Claude Code 或 Claude Desktop 中连接并测试。

要把本地 MCP 服务器连接到 Claude Code，请运行 `claude mcp add <name> <command> [args...]`。

要把本地 MCP 服务器或 DXT 连接到 Claude Desktop，请分别进入 Settings > Developer 或 Settings > Extensions。

工具也可直接传入 Anthropic API 调用中进行程序化测试。

生成评估任务

基于早期原型，Claude Code 可以快速探索工具并创建几十组提示词-响应对。提示词应来自真实场景，并基于真实数据源与服务（例如内部知识库和微服务）。我们建议避免过于简化或表面的“沙箱”环境，这类环境复杂度不足，无法有效压测工具。强评估任务可能需要多次工具调用，甚至几十次。

以下是一些高质量任务示例：

客户 Sarah Chen 刚提交取消请求。请准备挽留方案，并确定：(1) 她离开的原因；(2) 最有吸引力的挽留方案；(3) 在提出方案前需要关注的风险因素。

以下是一些较弱任务示例：

下周给 jane@acme.corp 安排会议。

在支付日志中搜索 `purchase_complete` 和 `customer_id=9182`。

找到客户 ID 45892 的取消请求。

每个评估提示词都应配对一个可验证响应或结果。验证器可以很简单，比如将真实答案与采样响应做精确字符串比对；也可以更高级，比如让 Claude 评审响应。避免过严验证器因格式、标点或同义表达差异而拒绝正确答案。

对于每组提示词-响应，你还可以选择指定期望智能体调用的工具，以衡量其是否真正理解各工具用途。但由于正确解法可能不止一种，尽量避免过度规定或对单一路径过拟合。

运行评估

我们建议通过直接 LLM API 调用以程序化方式运行评估。采用简单智能体循环（while 循环包装交替的 LLM API 和工具调用）：每个评估任务一个循环。每个评估智能体只接收一个任务提示词和你的工具。

在评估智能体的系统提示中，我们建议让智能体不仅输出结构化响应块（用于验证），还输出推理与反馈块。让这些块出现在工具调用与响应块之前，可能通过触发 chain-of-thought（CoT）行为提升 LLM 的有效智能。

如果你用 Claude 运行评估，也可以直接开启 interleaved thinking，获得类似“开箱即用”能力。这有助于你定位智能体为何调用或不调用某些工具，并突出工具描述与规格中待改进的具体区域。

除总体准确率外，我们还建议采集其他指标，如单次工具调用与任务总耗时、工具调用总数、总 token 消耗以及工具错误。追踪工具调用还能揭示智能体常见工作流，并提供工具整合机会。

分析结果

智能体是你发现问题和获得反馈的好伙伴，它们可以从互相矛盾的工具描述、低效实现到令人困惑的 schema 等方面给出建议。但要注意：智能体在反馈和响应中“没说出来的内容”往往比说出来的更重要。LLM 不总会把真实意图说全。

观察智能体在哪些地方卡住或困惑。阅读评估智能体的推理与反馈（或 CoT）识别粗糙边缘。审查原始转录（包括工具调用与工具响应），捕捉那些未在 CoT 中显式描述的行为。要“读懂字里行间”；记住评估智能体并不一定知道正确答案和策略。

分析你的工具调用指标。若有大量冗余调用，可能说明需要调整分页或 token 上限参数；若大量参数无效错误，可能说明工具描述应更清晰或示例应更好。我们上线 Claude web search 工具时发现，Claude 会不必要地在查询参数后追加 2025，导致搜索偏置和性能下降（我们通过优化工具描述把它引导回正确方向）。

与智能体协作

你甚至可以让智能体替你分析结果并改进工具。只需拼接评估智能体的转录并粘贴给 Claude Code。Claude 很擅长分析转录并一次性重构大量工具，例如在新增改动后保持工具实现与描述的一致性。

事实上，本文大部分建议都来自我们用 Claude Code 反复优化内部工具实现的过程。我们的评估建立在内部工作空间之上，完整映射了内部工作流复杂度，包括真实项目、文档和消息。

我们依赖留出测试集来确保不会对“训练集”评估过拟合。这些测试集显示，即便超越了所谓“专家级”工具实现，我们仍能继续挖掘性能提升空间，无论这些工具是研究员手写还是 Claude 生成。

编写高效工具的原则

本节我们把经验提炼为几条编写高效工具的指导原则。

为智能体选择正确工具

工具更多并不总意味着效果更好。我们常见的错误是：工具只是把现有软件功能或 API 端点简单包了一层，并未考虑它是否真正适合智能体。这是因为智能体与传统软件有不同“可供性”（affordances）——也就是它们感知可执行动作的方式不同。

LLM 智能体的“上下文”是有限的（即一次可处理信息量受限），而计算机内存便宜且充足。以通讯录找联系人为例，传统程序可以高效逐条处理联系人列表，逐个检查。

但如果 LLM 智能体调用的工具返回“全部联系人”，然后它必须逐 token 通读，就会把宝贵上下文浪费在无关信息上（想象你找联系人时只能逐页从头读到尾，这是暴力搜索）。更好也更自然的方式（对智能体和人都一样）是先跳到相关页面（例如按字母定位）。

我们建议先构建少量经过深思熟虑、面向高影响工作流且与评估任务匹配的工具，再逐步扩展。以通讯录为例，你可能应实现 `search_contacts` 或 `message_contact`，而不是 `list_contacts`。

工具可以整合功能，在内部处理多个离散操作（或 API 调用）。例如，工具可在响应中补充关联元数据，或将常见链式多步骤任务封装为一次工具调用。

示例：

与其实现 `list_users`、`list_events` 和 `create_event`，不如实现 `schedule_event`，让它完成可用时间查找并直接排期。

与其实现 `read_logs`，不如实现 `search_logs`，只返回相关日志行和周边上下文。

与其实现 `get_customer_by_id`、`list_transactions` 和 `list_notes`，不如实现 `get_customer_context`，一次性汇总客户近期且相关的信息。

确保你构建的每个工具都有清晰、独立目的。工具应让智能体像人类一样（在同等资源下）拆解并解决任务，同时减少本会被中间输出占用的上下文。

工具过多或功能重叠也会分散智能体注意力，降低策略效率。谨慎、有选择地规划“该构建什么工具（以及不构建什么）”会带来很高回报。

给工具做命名空间

你的 AI 智能体可能会接入几十个 MCP 服务器和上百个工具，包括其他开发者提供的工具。当工具功能重叠或目的模糊时，智能体很容易困惑该用哪个。

命名空间（把相关工具按公共前缀分组）有助于划定大量工具间边界；有些 MCP 客户端会默认这么做。例如按服务命名空间（如 `asana_search`、`jira_search`）及按资源命名空间（如 `asana_projects_search`、`asana_users_search`）都能帮助智能体在正确时机选择正确工具。

我们发现，前缀式与后缀式命名空间在工具使用评估中的影响并不小。影响因 LLM 而异，建议根据你自己的评估选择命名方案。

智能体可能会调错工具、用错参数、调用数量过少，或错误处理工具响应。通过有选择地实现那些名称反映任务自然拆分的工具，你可以同时减少加载到智能体上下文中的工具及描述数量，并把一部分智能体计算从上下文卸载到工具调用本身，从而降低整体出错风险。

从工具返回有意义的上下文

同理，工具实现应注意只向智能体返回高信号信息。应优先上下文相关性而非“全能性”，避免低层技术标识（如 `uuid`、`256px_image_url`、`mime_type`）。像 `name`、`image_url`、`file_type` 这类字段更可能直接影响智能体后续动作和响应。

与晦涩标识相比，智能体通常更擅长处理自然语言名称、术语或标识。我们发现，仅把任意字母数字 UUID 解析成语义更明确、可解释的文本（甚至 0-indexed ID 方案），就能明显提升 Claude 在检索任务中的精度并减少幻觉。

在某些场景中，智能体需要同时处理自然语言和技术标识输出，哪怕只是为触发后续工具调用（例如 `search_user(name='jane')` → `send_message(id=12345)`）。你可以通过在工具中暴露一个简单 `response_format` 枚举参数来兼顾两者，让智能体控制工具返回“concise”或“detailed”响应（见下图）。

为 token 效率优化工具响应

优化上下文质量很重要。但优化工具响应返回给智能体的上下文数量同样重要。

我们建议对可能占用大量上下文的工具响应，结合分页、范围选择、过滤和/或截断，并给出合理默认参数。以 Claude Code 为例，我们默认把工具响应限制在 25,000 tokens。我们预计智能体有效上下文长度会随时间增长，但对上下文高效率工具的需求不会消失。

如果你选择截断响应，一定要用有帮助的说明来引导智能体。你可以直接鼓励更省 token 的策略，例如针对知识检索任务，做多次小而精准搜索，而不是一次宽泛搜索。同样，当工具调用报错（例如输入校验失败）时，你可以通过提示词工程让错误响应清晰传达可执行改进建议，而非只给晦涩错误码或堆栈。

对工具描述做提示词工程

现在来到提升工具效果最有效的方法之一：对工具描述和规格做提示词工程。因为这些内容会加载进智能体上下文，它们共同决定智能体是否会形成高效工具调用行为。

编写工具描述和规格时，想象你在向团队新同事解释工具。把你可能默认带入的背景（特定查询格式、小众术语定义、底层资源关系等）显式写出来。通过清晰描述输入输出（并用严格数据模型强制）来避免歧义。尤其是输入参数命名必须明确：与其用 `user`，不如用 `user_id`。

借助评估，你可以更有信心地衡量提示词工程效果。即便是工具描述的小幅优化，也可能带来显著提升。我们对工具描述做了精细改进后，Claude Sonnet 3.5 在 SWE-bench Verified 评测上取得了当时 SOTA，显著降低错误率并提升任务完成效果。

你可以在我们的 Developer Guide 中找到更多工具定义最佳实践。如果你在为 Claude 构建工具，也建议阅读工具如何被动态加载到 Claude 系统提示中。最后，如果你在为 MCP 服务器编写工具，可用工具注解标明哪些工具需要开放世界访问或会造成破坏性变更。

展望未来

要为智能体构建有效工具，我们需要把软件开发实践从可预测、确定性的模式，转向非确定性模式。

通过本文描述的迭代式、评估驱动流程，我们识别出一组稳定模式：有效工具具有明确且清晰的定义，谨慎使用智能体上下文，可在多样工作流中自由组合，并让智能体直观地解决真实世界任务。

未来，智能体与世界交互的具体机制会持续演化——从 MCP 协议更新到基础 LLM 升级。只要采用系统化、评估驱动的方法持续改进工具，我们就能确保随着智能体能力提升，它们使用的工具也同步进化。

面向 AI 智能体的有效上下文工程

发表于 2025 年 9 月 29 日

上下文是 AI 智能体的关键但有限资源。本文将探讨如何有效策划和管理驱动智能体的上下文。

在应用型 AI 过去几年里，关注点从提示词工程逐步转向一个新术语：上下文工程。用语言模型构建系统，越来越不是“找到正确词句”这么简单，而是回答更大的问题：“怎样的上下文配置最可能产生我们期望的模型行为？”

上下文是指从大语言模型（LLM）采样时包含的 token 集合。工程问题在于：在 LLM 固有约束下优化这些 token 的效用，从而稳定达成目标结果。要有效驾驭 LLM，往往需要以“上下文视角”思考：即在任意时刻考虑 LLM 可见的整体状态，以及该状态可能产生的行为。

本文将探讨上下文工程这门新兴技艺，并给出一个更精炼的心智模型，帮助构建可控且高效的智能体。

上下文工程 vs. 提示词工程

在 Anthropic，我们把上下文工程视为提示词工程的自然演进。提示词工程是为获得最佳结果而编写和组织 LLM 指令的方法（可见我们的文档总览和实用策略）。上下文工程则是：在 LLM 推理期间策划并维护最优 token（信息）集合的一系列策略，包括所有可能进入上下文窗口、且不止来自提示词的信息。

在 LLM 工程早期，提示词是 AI 工程工作的最大组成部分，因为除日常聊天外，多数用例都需要为单轮分类或文本生成任务优化提示词。正如术语本身所示，提示词工程的关注点主要是如何写好提示词，尤其系统提示。但当我们迈向更强大的智能体工程（跨多轮推理、覆盖更长时间范围）时，我们需要管理整个上下文状态的策略（系统指令、工具、Model Context Protocol（MCP）、外部数据、消息历史等）。

循环运行的智能体会持续生成可能与下一轮推理相关的数据，这些信息必须被周期性精炼。上下文工程，就是从这个持续演化的信息宇宙中，策划哪些内容进入有限上下文窗口的艺术与科学。

提示词工程 vs. 上下文工程

不同于“写一个提示词”这种离散任务，上下文工程是迭代过程；每次决定传给模型什么信息时，都会发生策划阶段。

为什么上下文工程对构建高能力智能体很重要

尽管 LLM 速度越来越快、可处理数据量越来越大，但我们观察到它们和人类一样，会在某个点出现注意力下降或困惑。类似“针堆里找针”基准研究揭示了 context rot：随着上下文窗口 token 增多，模型从该上下文中准确召回信息的能力会下降。

虽然不同模型退化曲线有差异，但这一特性在所有模型中都存在。因此，上下文必须被视作“边际收益递减”的有限资源。就像人类工作记忆有限，LLM 也有“注意力预算”，在解析大量上下文时持续消耗。每引入一个新 token，预算就会被消耗一部分，这使得精心策划可用 token 变得更重要。

这种注意力稀缺来自 LLM 的架构约束。LLM 基于 Transformer 架构，它允许上下文中每个 token 关注其他每个 token。这会产生 n 个 token 对应的 n² 两两关系。

随着上下文长度增加，模型捕捉这些两两关系的能力会被摊薄，在上下文大小与注意力聚焦之间形成天然张力。此外，模型的注意力模式来自训练数据分布，而短序列通常多于长序列。这意味着模型在上下文全局依赖上的训练经验更少，专门参数也更少。

位置编码插值等技术可让模型通过适配到原本训练的小窗口来处理更长序列，但对 token 位置理解会有一定退化。这些因素共同造成的是“性能梯度”而非“硬悬崖”：模型在长上下文下仍然很强，但在信息检索和长程推理上的精度可能低于短上下文表现。

基于这些现实，精心的上下文工程是构建高能力智能体的必需项。

有效上下文的解剖

既然 LLM 受限于有限注意力预算，好的上下文工程就是找到尽可能小、但高信号的 token 集合，以最大化期望结果概率。实践远比表述困难，但在下节我们会说明这条原则如何落到上下文各组件。

系统提示应极其清晰，且用简洁直接语言在合适抽象层级表达。这个“合适高度”是两种常见失败模式间的 Goldilocks 区间。一端是工程师在提示词里硬编码复杂脆弱逻辑来逼出精确智能体行为，这会造成脆弱性并增加长期维护复杂度；另一端是提示过于笼统高层，无法给 LLM 足够具体信号，或错误假设共享背景。最佳高度是在两者之间平衡：既足够具体以有效引导行为，又足够灵活以给模型强启发式。

在上下文工程过程中校准系统提示。

在光谱一端是脆弱的 if-else 硬编码提示，另一端是过于泛化或错误假设共享上下文的提示。

我们建议将提示词组织为清晰分区（如 `<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` 等），并使用 XML 标签或 Markdown 标题进行分隔；不过随着模型能力提升，具体格式的重要性可能会下降。

无论你如何组织系统提示，都应追求“能完整定义期望行为的最小信息集”。（注意最小不等于最短；你仍需提供足够信息确保智能体遵循目标行为。）最佳方式是先用可用最强模型测试一个最小提示，看任务表现，再根据初测失败模式补充清晰指令与示例。

工具让智能体与环境交互，并在工作过程中拉取新增上下文。由于工具定义了智能体与其信息/行动空间之间的契约，工具必须促进效率：既要返回 token 高效率信息，也要鼓励智能体形成高效率行为。

在《借助 AI 智能体为 AI 智能体写工具》一文中，我们讨论了如何构建让 LLM 易理解且功能重叠最小的工具。类似设计良好的代码库函数，工具应自包含、抗错误，并且用途极其清晰。输入参数也应具备描述性、无歧义，并发挥模型固有优势。

我们最常见失败模式之一是工具集合臃肿：覆盖过多功能，或让“该用哪个工具”出现模糊决策点。如果人类工程师都无法明确某场景该用哪个工具，就不应期待 AI 智能体做得更好。正如后文将讨论，策划“最小可用工具集”也会让长交互中的上下文维护与裁剪更可靠。

提供示例（few-shot prompting）是广为人知且我们仍强烈建议的最佳实践。但团队常把大量边缘情况一股脑塞进提示词，试图覆盖 LLM 应遵循的每条规则。我们不建议这么做。相反，建议策划一组多样且典型的示例，准确传达智能体期望行为。对 LLM 来说，示例就是“一图胜千言”。

我们对上下文各组件（系统提示、工具、示例、消息历史等）的总体建议是：深思熟虑，让上下文信息足够丰富，同时保持紧凑。接下来我们看运行时动态检索上下文。

上下文检索与智能体搜索

在《构建高效 AI 智能体》中，我们强调了基于 LLM 的工作流与智能体的差异。自那篇文章后，我们逐渐采用一个简洁定义：智能体就是 LLM 在循环中自主使用工具。

与客户共建过程中，我们看到行业正收敛到这一简洁范式。随着底层模型能力增强，智能体自治水平可持续提升：更聪明模型可独立穿越细腻问题空间并从错误中恢复。

我们也看到工程师对“如何为智能体设计上下文”的思路在变化。如今很多 AI 原生应用会在推理前使用某种基于嵌入的检索，提前给智能体提供重要上下文。随着领域向更智能体化方向演进，我们越来越多看到团队用“just in time”策略增强这类检索系统。

这种方法不再预处理全部相关数据，而是维护轻量标识符（文件路径、已存查询、网页链接等），并借助工具在运行时动态加载数据到上下文。Anthropic 的 Claude Code 就采用了这种方式，可对大型数据库执行复杂数据分析。模型可以编写定向查询、存储结果，并用 `head`、`tail` 等 Bash 命令分析大数据量，而无需把完整数据对象载入上下文。这与人类认知类似：我们通常不会记住全部信息语料，而是通过文件系统、收件箱、书签等外部组织与索引系统按需检索相关信息。

除存储效率外，这些引用的元数据还提供了高效调优行为的机制，无论是显式提供还是隐含直觉。对在文件系统中运行的智能体来说，`tests` 文件夹下名为 `test_utils.py` 的文件，与 `src/core_logic/` 下同名文件所暗示的用途不同。目录层级、命名约定、时间戳都提供了重要信号，帮助人和智能体理解何时如何使用信息。

让智能体自主导航和检索数据还能实现 progressive disclosure，也就是通过探索逐步发现相关上下文。每次交互都会产生可指导下一决策的新上下文：文件大小暗示复杂度，命名约定提示用途，时间戳可作为相关性代理。智能体可层层构建理解，只在工作记忆中保留必要信息，并用记笔记策略做额外持久化。这种自管理上下文窗口让智能体聚焦相关子集，而不是淹没在穷尽但可能无关的信息里。

当然，这里有取舍：运行时探索比检索预计算数据更慢。不仅如此，还需要有主见且精细的工程设计，确保 LLM 拥有正确工具与启发式去高效导航信息空间。缺乏正确引导时，智能体可能误用工具、走入死胡同，或错过关键信息，从而浪费上下文。

在某些场景，最有效的智能体可能采用混合策略：一部分数据前置检索提速，另一部分按需自主探索。自治“合适边界”取决于任务。Claude Code 就使用了这种混合模型：`CLAUDE.md` 文件会先直接放入上下文，而 `glob`、`grep` 等原语让它能在环境中导航并按需检索文件，有效绕开索引陈旧和复杂语法树问题。

这种混合策略可能更适合内容动态性较低的场景，例如法律或金融。随着模型能力提升，智能体设计会更趋向“让智能模型做智能决策”，并逐步减少人工策划。鉴于该领域进展极快，“先做能工作的最简单方案”很可能仍是我们给基于 Claude 构建智能体团队的最佳建议。

面向长时任务的上下文工程

长时任务要求智能体在动作序列中保持连贯性、上下文一致性与目标导向行为，而这些序列的 token 数会超出 LLM 上下文窗口。对于持续数十分钟到数小时的任务（如大代码库迁移或综合研究项目），智能体需要专门技术来绕开上下文窗口大小限制。

等待更大的上下文窗口看起来是显而易见的策略。但在可预见未来，任何大小的上下文窗口都很可能受到上下文污染和信息相关性问题影响，至少在追求最强智能体性能的场景里如此。为让智能体在长时范围内有效工作，我们发展了几种直接应对这些约束的技术：压缩（compaction）、结构化记笔记以及多智能体架构。

压缩（Compaction）

压缩是指：当对话接近上下文窗口上限时，对其内容进行摘要，并以该摘要重新开启新上下文窗口。压缩通常是上下文工程中提升长期连贯性的第一道杠杆。其核心是高保真提炼上下文窗口内容，使智能体能在最小性能退化下继续工作。

以 Claude Code 为例，我们通过把消息历史交给模型，让其总结并压缩关键细节。模型会保留架构决策、未解决 bug 和实现细节，同时丢弃冗余工具输出或消息。随后智能体可以基于这个压缩上下文，加上最近访问的 5 个文件继续工作。用户获得连续体验，而不必担心上下文窗口限制。

压缩的艺术在于“保留什么、丢弃什么”的选择。过于激进的压缩可能丢失细微但关键的上下文，而其重要性往往在更晚才显现。对实现压缩系统的工程师，我们建议在复杂智能体轨迹上精细调参提示词。先最大化召回，确保压缩提示能捕获轨迹里所有相关信息；再迭代提升精度，去掉多余内容。

一个容易下手的冗余内容示例是清理工具调用与结果：某工具在历史深处调用过后，智能体为什么还需要再次看到原始结果？最安全、最轻量的压缩形式之一就是清理工具结果，这一能力最近已作为 Claude Developer Platform 功能上线。

结构化记笔记（Structured note-taking）

结构化记笔记，也叫智能体记忆（agentic memory），是指智能体定期把笔记写入上下文窗口外的持久记忆。这些笔记会在后续时刻被拉回上下文窗口。

这种策略以很小开销提供持久记忆。就像 Claude Code 创建待办清单，或你的自定义智能体维护 `NOTES.md` 文件，这个简单模式让智能体在复杂任务中持续跟踪进度，保留关键上下文和依赖，否则这些信息会在几十次工具调用后丢失。

Claude 玩宝可梦展示了记忆如何在非编码场景改变智能体能力。智能体能在数千个游戏步骤中保持精确计数，例如追踪目标“过去 1,234 步我一直在 1 号道路训练宝可梦，皮卡丘已从目标 10 级中提升了 8 级”。即使没有被提示记忆结构，它也会形成已探索区域地图，记住已解锁关键成就，并维护战斗策略笔记，学习不同对手下最有效攻击方式。

上下文重置后，智能体会读取自己的笔记，继续多小时训练序列或地牢探索。这种跨摘要步骤的一致性，使得长时策略成为可能；若仅依赖 LLM 上下文窗口保存全部信息，这些策略几乎无法实现。

在 Sonnet 4.5 发布中，我们在 Claude Developer Platform 公开测试版发布了 memory 工具，通过基于文件的系统让上下文窗口外信息的存储与查询更容易。这让智能体可以随时间积累知识库、跨会话维护项目状态，并在无需把所有内容放入上下文的情况下引用历史工作。

子智能体架构（Sub-agent architectures）

子智能体架构提供了绕开上下文限制的另一条路径。不同于单个智能体试图跨整个项目维护状态，专门子智能体可以在干净上下文窗口中处理聚焦任务。主智能体负责高层计划与协调，子智能体执行深度技术工作或使用工具寻找相关信息。每个子智能体可能进行大规模探索，消耗数万甚至更多 token，但只返回其工作的浓缩摘要（通常 1,000-2,000 tokens）。

这种方式实现了清晰的关注点分离：详细搜索上下文留在子智能体内部隔离，主智能体专注于综合和分析结果。这个模式在《How we built our multi-agent research system》中有讨论，并在复杂研究任务上相较单智能体有显著提升。

这些方法的选择取决于任务特性。例如：

压缩适合需要大量来回交互的任务，可维持对话流畅性；

记笔记适合具有清晰里程碑的迭代开发；

多智能体架构适合并行探索收益明显的复杂研究与分析任务。

即使模型持续进步，如何在长交互中维持连贯性，仍会是构建更有效智能体的核心挑战。

结论

上下文工程代表了我们用 LLM 构建系统方式的根本转变。随着模型能力提升，挑战不再只是写出完美提示词，而是在每一步精心策划哪些信息应进入模型有限注意力预算。无论你是在为长时任务实现压缩、设计高 token 效率工具，还是让智能体按需探索环境，指导原则始终不变：找到最小的高信号 token 集合，以最大化你期望结果的概率。

随着模型进步，我们提出的技术会继续演化。我们已经看到，更聪明的模型需要更少规定性工程，让智能体具备更高自治性。但即使能力持续扩展，把上下文当作珍贵且有限资源，依然是构建可靠高效智能体的核心。

现在就可在 Claude Developer Platform 开始实践上下文工程，并通过我们的记忆与上下文管理 cookbook 获取实用建议和最佳实践。

用于长时运行智能体的高效 harnesses

发表于 2025 年 11 月 26 日

智能体在跨多个上下文窗口工作时仍面临挑战。我们从人类工程师那里汲取灵感，为长时运行智能体构建了更高效的 harness。

随着 AI 智能体能力增强，开发者越来越希望它们承担跨越数小时甚至数天的复杂任务。然而，让智能体在多个上下文窗口中持续稳定推进仍是开放问题。

长时运行智能体的核心挑战是：它们必须在离散会话中工作，而每个新会话都从“没有前情记忆”开始。想象一个软件项目由轮班工程师接力，新上岗工程师对上一班发生了什么毫无记忆。由于上下文窗口有限，且多数复杂项目无法在单窗口内完成，智能体需要一种机制来跨会话衔接。

我们开发了一个双管齐下方案，使 Claude Agent SDK 能跨多个上下文窗口高效工作：首次运行时由 initializer agent 搭建环境；后续由 coding agent 在每个会话里做增量推进，同时为下个会话留下清晰工件。配套 quickstart 中可找到代码示例。

长时运行智能体问题

Claude Agent SDK 是一个强大的通用智能体 harness，擅长编码，也适合其他需要模型用工具收集上下文、规划和执行的任务。它具备上下文管理能力（如 compaction），可让智能体在不耗尽上下文窗口的情况下持续工作。理论上，在这套机制下，智能体应能无限期地继续产生有价值工作。

但 compaction 并不充分。开箱即用情况下，即使是 Opus 4.5 这种前沿编码模型，在 Claude Agent SDK 上跨多个上下文窗口循环运行，如果只给“build a clone of claude.ai”这类高层提示，也难以做出生产级 web 应用。

Claude 的失败主要表现为两种模式。第一，智能体倾向于一次做太多，实质上想“一把梭”完成应用。结果常是实现中途耗尽上下文，让下个会话接手一个半成品且无文档的功能。智能体随后不得不猜测发生了什么，并花大量时间先把基础应用重新跑通。即使有 compaction 也会发生，因为 compaction 传给下一个智能体的指令并不总是足够清晰。

第二种失败通常出现在项目后期。已有一些功能后，后续智能体实例环顾一圈，看到已有进展，就直接宣布任务完成。

这可拆成两部分问题。第一，需要搭建一个初始环境，为给定提示所需全部功能打基础，让智能体能按步骤、按功能逐步推进。第二，应提示每个智能体做增量推进，同时在会话结束时保持环境“干净状态”。所谓“干净状态”，是指可直接合入主分支的代码状态：无重大 bug、代码整洁且文档充分，开发者通常可直接开始新功能，而无需先清理无关烂摊子。

在内部实验中，我们通过两段式方案解决这些问题：

Initializer agent：第一个智能体会话使用专门提示词，让模型搭建初始环境：`init.sh` 脚本、记录智能体工作日志的 `claude-progress.txt` 文件，以及展示新增文件的初始 git commit。

Coding agent：后续每个会话都要求模型做增量推进，并留下结构化更新。1

核心洞察是：要让智能体在新鲜上下文窗口启动时快速理解当前工作状态。我们通过 `claude-progress.txt` 加上 git 历史实现了这一点。这些实践灵感来自高效软件工程师的日常工作方式。

环境管理

在更新后的 Claude 4 提示词指南中，我们分享了多上下文窗口工作流的最佳实践，其中包括“首个上下文窗口使用不同提示词”的 harness 结构。这个“不同提示词”会要求 initializer agent 搭建环境，并包含未来 coding agents 高效工作所需的全部必要上下文。这里我们更深入介绍这类环境的几个关键组件。

功能列表

为解决智能体一把梭做应用或过早判断项目完成的问题，我们提示 initializer agent 写一个完整功能需求文件，基于用户初始提示做扩展。在 claude.ai 克隆示例中，这意味着 200+ 功能，例如“用户可打开新聊天、输入查询、按回车并看到 AI 响应”。这些功能初始都标为“failing”，以便后续 coding agents 清晰理解“完整功能”应是什么样。

我们提示 coding agents 仅通过修改 `passes` 字段状态来编辑该文件，并用强措辞约束，例如“删除或修改测试是不可接受的，因为这可能导致功能缺失或 bug。”经过实验，我们最终选用 JSON，因为模型相比 Markdown 更不容易不当修改或覆盖 JSON 文件。

增量推进

在这套初始环境脚手架下，下一轮 coding agent 被要求一次只处理一个功能。事实证明，这种增量方式对纠正“做太多”的倾向至关重要。

即便采用增量方式，模型在代码变更后仍必须把环境留在干净状态。我们的实验发现，最能触发这一行为的方法是要求模型把进度提交到 git（使用描述性 commit message），并在进度文件写总结。这让模型可以使用 git 回滚坏改动，并恢复代码库可工作状态。

这些做法也提升了效率，因为它们消除了智能体“猜发生了什么并花时间重修基础应用”的必要。

测试

我们观察到的最后一个主要失败模式是：Claude 在未充分测试的情况下把功能标记为完成。若无显式提示，Claude 往往会做代码修改，也会运行单元测试或对开发服务器执行 curl 测试，但无法识别功能并未端到端可用。

在构建 web 应用场景中，一旦显式提示 Claude 使用浏览器自动化工具并按人类用户方式做测试，Claude 在端到端验证方面表现总体不错。

给 Claude 配备这类测试工具显著提升了性能，因为智能体能够发现并修复仅靠读代码不明显的 bug。

仍有一些问题存在，例如 Claude 视觉能力和浏览器自动化工具的限制，使其难以识别所有类型 bug。比如 Claude 无法通过 Puppeteer MCP 看到浏览器原生 alert modal，依赖这些 modal 的功能因此更容易出错。

快速进入状态

在上述机制就位后，每个 coding agent 会被提示先执行一组步骤来“摸清情况”，有些很基础但依然有帮助：

运行 `pwd` 查看当前工作目录。你只能编辑该目录中的文件。

阅读 git 日志和进度文件，快速了解近期完成了什么。

阅读功能列表文件，选择尚未完成且优先级最高的功能开始处理。

这种做法在每个会话都能为 Claude 节省一些 token，因为它不用再摸索如何测试代码。同时，也建议让 initializer agent 写一个可启动开发服务器的 `init.sh`，并在实现新功能前先跑一个基础端到端测试。

在 claude.ai 克隆案例中，这意味着智能体总会先启动本地开发服务器，并用 Puppeteer MCP 发起新聊天、发送消息并接收回复。这样 Claude 能快速识别应用是否处于损坏状态，并立即修复既有 bug。如果它反而先实现新功能，通常会让问题更糟。

基于以上机制，一个典型会话通常会以如下助手消息开始：

[Tool Use] <bash - pwd>

[Tool Use] <read - claude-progress.txt>

[Tool Use] <read - feature_list.json>

[Assistant] Let me check the git log to see recent work.

[Tool Use] <bash - git log --oneline -20>

[Assistant] Now let me check if there's an init.sh script to restart the servers.

<Starts the development server>

未来工作

这项研究展示了一组可行方案：在长时运行智能体 harness 中，让模型跨多个上下文窗口做增量推进。不过仍有开放问题。

最重要的是，目前仍不清楚：单一通用 coding agent 是否在跨上下文场景里最佳，还是多智能体架构能获得更好表现。看起来很合理的是，测试智能体、质量保证智能体、代码清理智能体等专门角色，可能会在软件开发生命周期各子任务上做得更好。
